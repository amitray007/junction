### Security-review outcomes (post-build hardening)

The `ce-security-reviewer` confirmed the config is clean (least-privilege permissions, no `pull_request_target`/injection, no npm-publish path). Applied hardening:
- **SHA-pinned** the two security-sensitive third-party actions — `changesets/action` (runs with write token) and `gitleaks/gitleaks-action` (scans history) — to full commit SHAs with `# vX` comments. Dependabot's github-actions updater still bumps them. First-party `actions/*`/`pnpm/*` stay on tags (right-sized).
- **`no-publish-guard` CI job** (in `security.yml`) — greps `ci.yml`+`release.yml` and fails if a real `npm/pnpm/changeset publish` path or `NODE_AUTH_TOKEN`/`NPM_TOKEN` is ever introduced. Locks in the publish-disabled invariant.

**Accepted tradeoff (documented):** the `main` ruleset keeps `required_approving_review_count: 0`. This is a **solo-maintainer** repo — GitHub doesn't let you approve your own PR, so requiring ≥1 approval would make merging impossible. The PR gate + required status checks still apply. Revisit (raise to ≥1, add code-owner review) if/when junction gains co-maintainers.

**Follow-up (not blocking 0.75):** add a scheduled CI check that diffs the live ruleset (`gh api .../rulesets/17989650`) against the committed `.github/rulesets/main.json` to catch drift. Tracked for the security increment.

## Part 2 — Implementation

### Step 1 — `.github/workflows/ci.yml`

Pin: `actions/checkout@v5`, `pnpm/action-setup@v4` (with `cache: true`), `actions/setup-node@v4` (no `cache:` — action-setup owns it). Two jobs:

```yaml
name: CI
on:
  push: { branches: [main] }
  pull_request:
concurrency:
  group: ci-${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
permissions:
  contents: read
jobs:
  verify:
    name: verify (node ${{ matrix.node }})
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        node: [20, 22]
    steps:
      - uses: actions/checkout@v5
      - uses: pnpm/action-setup@v4
        with: { cache: true }
      - uses: actions/setup-node@v4
        with: { node-version: ${{ matrix.node }} }
      - run: pnpm install --frozen-lockfile
      - run: pnpm verify
  quality:
    name: quality
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
      - uses: pnpm/action-setup@v4
        with: { cache: true }
      - uses: actions/setup-node@v4
        with: { node-version: 22 }
      - run: pnpm install --frozen-lockfile
      - run: pnpm run quality
```

Add a root `"quality"` script to `package.json` — placeholder now: `"quality": "pnpm audit --audit-level=high || true"` (inc 1.5 expands it to `knip && depcruise && syncpack:check && dup && pnpm audit ...`).

### Step 2 — `.github/workflows/release.yml`

`changesets/action@v1.9.0`, full history, **`publish: pnpm changeset tag`** (tags + GitHub releases, never `npm publish`):

```yaml
name: Release
on:
  push: { branches: [main] }
concurrency: { group: release-${{ github.workflow }}-${{ github.ref }} }
permissions:
  contents: write
  pull-requests: write
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
        with: { fetch-depth: 0 }
      - uses: pnpm/action-setup@v4
        with: { cache: true }
      - uses: actions/setup-node@v4
        with: { node-version: 22 }
      - run: pnpm install --frozen-lockfile
      - uses: changesets/action@v1.9.0
        with:
          version: pnpm changeset version
          publish: pnpm changeset tag
          createGithubReleases: true
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

> **Smoke-test caveat:** `changeset tag` is the documented no-publish substitute, but `changesets/action`'s GitHub-release creation parses the publish command's `New tag:` output. Confirm GitHub Releases actually appear on a test tag; if parsing ever misses, fall back to `gh release create` keyed off the tags. Treat this as the one thing to verify, not assume.

### Step 3 — Changesets config

`pnpm add -Dw @changesets/cli @changesets/changelog-github`, then `pnpm changeset init`, then overwrite `.changeset/config.json`:

```json
{
  "$schema": "https://unpkg.com/@changesets/config@3.1.1/schema.json",
  "changelog": ["@changesets/changelog-github", { "repo": "<owner>/junction" }],
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "restricted",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": [],
  "privatePackages": { "version": true, "tag": true }
}
```

Replace `<owner>` with the real GitHub owner (ask the user if unknown). **`privatePackages: {version:true, tag:true}`** is load-bearing: it versions+tags `private: true` packages without publishing. Ensure every package keeps `"private": true` (the root already is; packages from inc 1 must be too).

### Step 4 — `.github/dependabot.yml`

```yaml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule: { interval: "weekly", day: "monday" }
    open-pull-requests-limit: 10
    groups:
      minor-and-patch: { update-types: ["minor", "patch"] }
      types: { patterns: ["@types/*"] }
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule: { interval: "weekly" }
    groups:
      actions: { patterns: ["*"] }
```

Then (if on GitHub + authed): enable Dependabot **alerts + security updates** in repo Settings → Security (or `gh api`). Note in the doc that security PRs fire immediately, independent of this schedule.

### Step 5 — `main` ruleset

Configure via `gh api` for `amitray007/junction` (rulesets aren't committed config — but export a snapshot). Settings: target `main`, Active; restrict deletions; block force-push; require PR (approvals: **0** — solo; require conversation resolution); require status checks `verify (node 20)`, `verify (node 22)`, `quality` with branches-up-to-date. **Do NOT** require linear history (merge-commit workflow) and **do NOT** require signed commits (deferred). After applying, export:

```bash
gh api repos/<owner>/junction/rulesets > /tmp/rs.json   # find the id
gh api repos/<owner>/junction/rulesets/<id> > .github/rulesets/main.json
```

Commit `.github/rulesets/main.json` as a reviewable, restorable snapshot. Document that GitHub does **not** auto-sync it — edits happen via UI/API, then re-export. If the repo isn't on GitHub yet, **write the intended ruleset JSON to `.github/rulesets/main.json` as the spec** and leave a note to apply it once the repo exists.

### Step 6 — `LICENSE` + license fields

- Fetch the **full AGPL-3.0 text** from `https://www.gnu.org/licenses/agpl-3.0.txt` → repo-root `LICENSE`.
- Add `"license": "AGPL-3.0-only"` to root `package.json` (and to each package's `package.json` as they're created — note for inc 1).

### Step 7 — `.editorconfig`

Basic cross-editor consistency (UTF-8, LF, final newline, 2-space, trim trailing ws) — aligned with `biome.json` (2-space, double quotes). Keep minimal.

### Step 8 — PR-title lint — SKIP

User merges via **merge commit**, not squash, so PR-title lint adds no value. **Skip this step.**

### Step 9 — Verify & commit

- `pnpm install` (adds changeset deps); `pnpm verify` green.
- Validate all workflow YAML (e.g. `gh workflow view` or a yaml lint).
- Confirm `pnpm changeset` runs.
- `git add -A && git commit` (pre-commit runs verify).
- If on GitHub: push a throwaway branch + PR to confirm CI actually triggers and the `verify`/`quality` checks appear — then clean up.
