# Method File 00.5 — Rules & Enforcement (Increment 0.5)

> **Lands after increment 0, before increment 1.** This is the "we never write bad code / never ship broken code" layer: the per-language **rules** plus the **mechanical enforcement** (verify script + hooks) that makes the rules unskippable. Realizes design spec §5b and §7.
>
> **Written directly by the orchestrator** (prose rule docs + small config/hook files — no implementation logic). The one judgment call: the hook scripts are tiny shell/JS; write them directly and test them on a throwaway edit/commit.

---

## Part 1 — Spec (what & why)

### Goal

Before any package code exists, establish: (a) `docs/rules/` — the enforceable coding guardrails per language; (b) the QA loop (`pnpm verify`) and the hooks that run it, so an agent literally cannot ship broken code or violate boundaries.

### Deliverables

| # | Artifact | Path | Notes |
|---|---|---|---|
| 1 | Rules index | `docs/rules/README.md` | how rules are enforced (skills cite, hooks check, agents audit) |
| 2 | TypeScript rules | `docs/rules/typescript.md` | neverthrow Result, typed errors, no `any`, Zod at boundaries, file size, naming, ESM hygiene |
| 3 | Testing rules | `docs/rules/testing.md` | Vitest conventions; "QA-able per change"; behavior-not-implementation |
| 4 | Performance rules | `docs/rules/performance.md` | no sync I/O in core/server, async logging, lazy imports, bench only hot paths |
| 5 | Security rules | `docs/rules/security.md` | credential plaintext never logged/persisted/returned; banned APIs; secrets-in-errors forbidden |
| 6 | Verify script + Biome | root `package.json` `verify` script (placeholder until inc.1 has packages) + `biome.json` | `verify` = typecheck + biome + vitest |
| 7 | lefthook config | `lefthook.yml` | pre-commit runs `pnpm verify` |
| 8 | Claude Code hooks | `.claude/settings.json` | per-edit biome; pre-commit verify; PreToolUse boundary guard |
| 9 | Boundary-guard script | `.claude/hooks/boundary-guard.mjs` | blocks banned imports/patterns in a diff |

### Proof of done

- `docs/rules/` exists with all five files; each rule is concrete and checkable.
- `pnpm verify` exists (even if it no-ops gracefully before packages exist) and is documented as the gate.
- Hooks fire: a throwaway edit triggers Biome; a deliberately-broken commit is blocked by the pre-commit verify; an edit introducing a banned import (e.g. `import x from "../cli"` inside `core`, or `node:vm`) is blocked by the boundary guard.
- `CLAUDE.md` and the design spec already reference `docs/rules/` and §5b — confirm no contradiction.
- Committed to git.

### Out of scope

Package code, real tsconfig/build wiring beyond what `verify` needs to exist — those are increment 1. Here we set up the *enforcement frame* so increment 1's code is governed immediately.

## Part 2 — Implementation (step by step)

### Step 1 — `docs/rules/README.md`

Explain the enforcement model: rules are **cited** by the `junction-clean-code` skill, **checked mechanically** by hooks (the subset that's greppable/automatable), and **audited** by the review agents (`junction-clean-code-reviewer`, `junction-package-boundary`, and the stubs). State precedence: a hook block is a hard stop; an agent finding is advisory-but-expected-to-be-resolved. List the rule files.

### Step 2 — `docs/rules/typescript.md`

Concrete TS rules (each phrased as a checkable MUST/MUST-NOT):
- Fallible operations return **neverthrow `Result<T, E>`**; no bare `throw` across module/package boundaries. Domain errors are **discriminated unions** (`type CredentialError = DecryptionError | KeyNotFoundError | StorageError`).
- `eslint-plugin-neverthrow`-style discipline: a returned `Result` must be consumed (no floating Results). (Enforced by review agent; lint rule if/when ESLint is added for this niche.)
- **No `any`, no non-null `!`** in `core`; `noUncheckedIndexedAccess` honored. Validate external input with **Zod** at boundaries.
- Resource cleanup via **`using` / `Symbol.asyncDispose`**, not ad-hoc try/finally where a disposable fits.
- **File size / single purpose:** if a file does two jobs or grows past readability, split it. One module = one clear responsibility.
- ESM import hygiene: explicit extensions where `nodenext` requires, no deep-importing another package's internals, respect the dependency direction.
- Intention-revealing names; no abbreviations that aren't domain terms.

### Step 3 — `docs/rules/testing.md`

- Vitest, tests colocated (`*.test.ts`) or in `__tests__`. Assert **behavior**, not internal calls.
- **Every change is QA-able:** it ships with at least one behavior test and passes `pnpm verify`. No "I'll test it later."
- Use `JUNCTION_HOME=<tmpdir>` to isolate filesystem state in tests.
- Credential/security paths get explicit negative tests (e.g. "plaintext never written to disk").

### Step 4 — `docs/rules/performance.md`

- **No `fs.*Sync` / `execSync`** in `core` or server paths (CLI init scripts may use sync). Blocks the event loop.
- Structured **async logging via pino** for machine/audit; consola for human CLI. Never log secrets.
- **Lazy-import heavy deps** inside the handler that needs them (keeps CLI cold start fast).
- `vitest bench` only for genuinely hot paths (credential encrypt/decrypt, MCP tool dispatch, sandbox spawn) — don't bench speculatively.

### Step 5 — `docs/rules/security.md`

- Credential **plaintext never** logged, persisted in the main DB, or returned over MCP. Only in memory during a call.
- Banned: `node:vm`/`vm2` as a sandbox, `eval` of untrusted input, secrets embedded in error messages.
- Secret backend abstraction (`CredentialStore`) is the only path to plaintext.

### Step 6 — `biome.json` + root `verify` script

- `biome.json`: recommended rules on; formatter on; organize-imports on. Add a rule (or a documented grep in the boundary guard) banning `fs.*Sync` in core/server globs.
- Root `package.json` scripts: `"verify": "tsc -b && biome check . && vitest run"` (before packages exist, make each step a graceful no-op or skip — document that it becomes real at increment 1). Also `"lint"`, `"format"`, `"test"`, `"test:related"` per spec §3.

### Step 7 — `lefthook.yml`

- `pre-commit`: run `pnpm verify` on staged changes; fail → block commit. Use lefthook's `{staged_files}` scoping.
- Keep it fast: format/lint on staged files; full verify as the gate.

### Step 8 — `.claude/settings.json` hooks

Wire the tiered hooks (spec §5b). Note: the **commit gate is lefthook** (Step 7), not a Claude Code hook — `.claude/settings.json` handles only per-edit format + the boundary guard:
- **PostToolUse** (Edit/Write): `biome check --write` on the edited file only (via `.claude/hooks/format-edited.mjs`). Fast.
- **PreToolUse** (Edit/Write): run `.claude/hooks/boundary-guard.mjs`; block (exit 2) on banned patterns.

### Step 9 — `.claude/hooks/boundary-guard.mjs`

A tiny Node script that reads the proposed edit / current diff and **blocks (exit 2)** if it detects:
- an import from `cli`/`web`/`mcp/*` *into* a `core` file,
- an HTTP-server/daemon lib imported in `core`,
- `node:vm` or `vm2` anywhere,
- `fs.*Sync`/`execSync` in `core`/server globs.
Print the violated rule + file so the agent can self-correct. Keep it dependency-free.

### Step 10 — Verify & commit

- Test each hook on a throwaway file: confirm Biome fixes formatting, the boundary guard blocks a banned import, and a broken commit is rejected.
- Remove the throwaway artifacts.
- `git add -A && git commit`.
