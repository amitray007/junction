## 1. What & why

This is the **setup / foundation** increment for the web UI. It does **not** add new functional
backends — it **re-establishes the web's design system and rebuilds the four surfaces on it**, so that
later increments (25 = platform mutations, 26 = profile/source mutations, a later keys increment, 29 =
audit) "just wire the functional work" onto a finished chassis. (Decided with the user: *"split 24.5 to
just the setup idea; inc 25 is the real work."*)

Three things ship:

1. **A fresh, Geist-grade design system** — a full rewrite of `docs/design/DESIGN.md` + the token layer
   (`packages/web/src/styles/app.css`) + the `ui/` primitives, replacing the inc-23 "instrument" system
   (which the user found slop-adjacent: side-accent nav bar, mono-eyebrows everywhere, 13px body,
   amber-glow). The new system is dark **and** light, Geist Sans/Mono, numbered-intent color tokens,
   blue accent for state, subtle shadows, Geist copy voice. Full spec in §3.
2. **The four surfaces rebuilt** in the new language: **Credentials (fully wired)**, **Dashboard (full —
   the single-endpoint "Connect an Agent" piece is the main new value)**, **Profiles (full read +
   route rows)**, **Platforms (lighter re-skin)**.
3. **A "Coming soon" pattern** on everything whose backend isn't built yet (recent activity, platform
   mutations, profile/route mutations, key management / the single-endpoint auth), so the whole *shape*
   exists truthfully without faking functionality.

### Non-goals (explicitly deferred — show as "Coming soon", build the wiring later)
- **No platform mutations** (Add/Edit/Delete platform) — inc 25.
- **No profile/source mutations** (New profile, Add route, Edit tool access, toggle) — inc 26.
- **No junction-keys / single-endpoint auth backend** — its own later increment. v1 *shows* the
  single-endpoint model + key→profile framing, but no key table, no endpoint auth, keys are "Coming soon".
  (Recorded in `docs/futures/revisit-when.md`; it intentionally revises the per-profile-endpoint invariant.)
- **No audit log / real analytics** — inc 29. "Recent activity" + per-agent usage are "Coming soon".
- **No live status probing** — inc 28. Badges stay **Configured**.

## 2. Hard invariants (load-bearing)

1. **Server-only-core boundary preserved** (inc 22/23/24). No `@junction/core` (or `better-sqlite3`/
   `@napi-rs/keyring`) in any client-reachable module. The `web:leakcheck` + `web:smoke` gates stay green.
   This is a UI/design increment — it should touch **zero** server/core logic except where noted (the
   Credentials surface reuses the **existing** inc-24 mutations; do not change them).
2. **Credentials stay metadata-only.** No secret/secretRef in any component, loader, or response. The
   existing `CredentialMeta` shape is the contract. The new design must not introduce any field that
   leaks a secret.
3. **No faked functionality.** Anything not backed by a real, wired server-fn is presented via the
   **`ComingSoon`** affordance (disabled control + a quiet "Coming soon" pill + a one-line CLI hint),
   never as a working-looking button that errors or no-ops silently. The ONLY mutations that actually
   run in v1 are the inc-24 credential add/rotate/delete.
4. **Light AND dark are first-class.** Every surface + primitive renders correctly in both themes
   (the existing theme toggle + `prefers-color-scheme` SSR-no-flash mechanism from inc 23 is reused;
   do not regress the no-flash behaviour — see `docs/futures/gotchas.md` SSR cookie note).
5. **a11y + anti-slop** (web.md + the new DESIGN.md): semantic HTML, visible focus ring on every
   interactive element, status = dot + text (never color-only), keyboard-reachable, reduced-motion-safe,
   tokens-only (no magic hex/px outside the `@theme` block), Geist copy voice. The design is authored
   **against** `docs/design/anti-ai-slop.md` — re-read it; do not reintroduce the tells.
6. **Every change ships QA-able** (`docs/behaviours/verify-the-artifact.md`): `pnpm verify` green (incl.
   `verify:web`: build + leakcheck + smoke), plus a `junction-web-verify` browser pass on **both themes**,
   plus component tests for each rebuilt surface + each new/changed primitive.

## 3. The new design system (authoritative)

The full rewritten design system **is already at `docs/design/DESIGN.md`** (it replaced the inc-23
"instrument" content during planning). The builder MUST implement to that file; the rationale + the
domain model it serves are in `docs/design/24.5-ux-foundation-notes.md`, and the anti-slop reference is
`docs/design/anti-ai-slop.md`. Summary of the decisions DESIGN.md encodes (read it for exact tokens/values):

- **Type:** Geist Sans (UI/headings/body) + Geist Mono (IDs/endpoints/namespaces/config/counts). Already
  vendored in `packages/web/src/styles/fonts/`. Body floor **14px**. ≤2 weights per view. Title Case
  labels/buttons, sentence-case body.
- **Color:** Geist numbered-intent tokens (gray ranks info; 100–300 bg / 400–600 border / 700–800 fill /
  900–1000 text). **One accent = blue** for state/links/focus/endpoint-identity. **Primary button = solid
  gray-1000 fill** (not blue). Dark = true black `#000`, light = true white; same token names; both equal.
  Exact light+dark hex are in DESIGN.md§Color (mirror them into `app.css`).
- **Space:** 4px base; **8 in-group / 16 between groups / 40 between sections**. Content col ≤960px,
  `scrollbar-gutter:stable`. Card padding 20–24. Radii **6** (controls/surfaces) / **12** (cards/menus).
- **Elevation:** subtle two-layer shadows (`--shadow-sm`/`--shadow-md`) — this **reverses** inc-23's
  "borders not shadows". Keep borders for hairline separation + shadow for card lift.
- **Motion:** 0/150/200/300ms; transform/opacity only; **never `transition: all`**; reduced-motion-safe.
- **Signature element:** the **route row** (`source → account → ns-chip · filter → on/off`) replaces the
  inc-23 live status rail. **Retire the status rail** (`ui/status-rail.tsx`) and the instrument wordmark
  affectations; the new wordmark is the `J` glyph + "Junction" (see DESIGN.md§App shell).
- **New primitive — `ComingSoon`:** a quiet pill (`--gray-100` bg, `--gray-700` text, "Coming soon") +
  helper to render a disabled control with the pill + a CLI hint. Used on every deferred action.

## 4. Implementation

> Order matters: design tokens → primitives → surfaces → gates. Rebuild on the **existing** route/ui/server
> files; do not rescaffold the app shell or the server layer.

### Phase A — design system: tokens + rules
- `docs/design/DESIGN.md` is already the new Geist-grade system (done in planning) — implement to it.
  `docs/rules/web.md` is **also already updated** (amber→blue, Departure-Mono retired, body ≥14px,
  Geist-grade subtle shadows allowed) so the review gate won't fight the new design — keep app.css in sync
  with both; do NOT reintroduce inc-23 rules.
- Rewrite `packages/web/src/styles/app.css` `@theme` + `:root`/dark token blocks to the new tokens. **Define
  EVERY token DESIGN.md references (tokens-only is a hard rule — no magic hex/px in components):** neutrals
  incl. **`--gray-600`**, the **blue** accent set, the **status** hues as named token pairs (`--status-ok-fg`,
  `--status-warning-fg`, `--status-error-fg`, `--status-noauth-fg`, etc. — mirror the existing `--status-*`
  naming, light+dark), the **`--shadow-sm`/`--shadow-md`** two-layer elevation values (light+dark), type scale,
  spacing (8/16/40 rhythm), radii (6/12). Keep the Geist `@font-face` blocks (fonts are **vendored raw `.woff2`**
  in `src/styles/fonts/` — there is NO `geist` npm dep; do not add one). Preserve the dark-mode mechanism
  (data-theme + prefers-color-scheme) and `scrollbar-gutter: stable`.
- **Full Departure-Mono removal** (it's retired; leaving it is the anti-slop "dead scaffolding" tell): remove
  the `@font-face`, the `--font-display` token, and the `.woff2` + OFL file usage; `wordmark.tsx` no longer
  uses it. (Keep `dropdown`/`dialog`/`tabs`/`tooltip`/`select`/`switch`/`checkbox` on **Radix** — web.md
  requires Radix primitives; restyle, don't hand-roll, so a11y doesn't regress.)
- **Keep primitives on Radix UI + cva** (web.md rule) — restyle to the new tokens; never hand-roll an
  interactive primitive that Radix provides.

### Phase B — ui/ primitives (rework in place; retire what the new system drops)
Update each to the new tokens + Geist component conventions (read `docs/design/DESIGN.md` §Components):
- **Keep + restyle:** `button` (Geist hierarchy: primary=gray-1000 fill, secondary, tertiary/ghost, error),
  `badge`/StatusBadge (dot+text, new status tokens, keep the **Configured** taxonomy), `card` (subtle shadow,
  12px radius, header+body, never nested), `table` (+ actions cell), `dialog`, `dropdown-menu`, `input`/`field`/
  `select`/`switch`/`checkbox` (the inc-24 form primitives — restyle, keep the a11y incl. the field
  aria-describedby/aria-invalid fix), `tabs`, `tooltip`, `separator`, `kbd`, `skeleton`, `states`
  (empty/loading/error), `sidebar` (plain-text destinations + counts, **no side-accent stripe** — active =
  gray-100 bg + gray-1000 text), `page-header`.
- **New:** `ComingSoon` (the pill + disabled-control-with-CLI-hint helper) — add to `ui/` + barrel + a test.
- **New:** `route-row` (the signature) — `source → account → ns-chip · filter → on/off`; + a test.
- **New:** `agent-config` block — renders the *intended* single-endpoint + key→profile model (endpoint in
  mono/blue, tabbed config, key→profile chips) **as a `ComingSoon` surface** (visually present, disabled,
  with the stdio CLI hint — see the Dashboard CRITICAL note). It must NOT present a working HTTP endpoint or
  a live Copy. Used on the Dashboard.
- **Retire:** `status-rail` (+ its test) and the instrument wordmark bits — replace `wordmark` with the new
  `J`-glyph + "Junction" lockup. Update `ui/index.ts` barrel + `__root.tsx` (remove the StatusRail zone).
- Every primitive renders light + dark; keep/adjust each `*.test.tsx`.

### Phase C — the 4 surfaces (routes)
Rebuild each route component on the new primitives. Loaders/server-fns are **unchanged** (reuse existing
`data.functions.ts` reads + `mutations.functions.ts` for credentials).

- **`routes/index.tsx` (Dashboard) — FULL.** Status line (running · `credentialStore` · `sandbox` from
  `DashboardData`) · **Connect an Agent** → **render the WHOLE block as a `ComingSoon` surface** (see the
  critical note below): show the intended single-endpoint + key→profile model visually, **disabled**, with a
  one-line CLI hint. · **At a Glance** (the `counts` as a small stat row, NOT a banner) · **Recent Activity**
  → **ComingSoon** (audit, inc 29).

  > ⚠️ **CRITICAL (review blocker) — do NOT render a working `http://localhost:<port>/mcp` endpoint or a
  > copy-pasteable HTTP config.** Junction's MCP server is **stdio-only** (`packages/mcp/server/src/serve.ts`
  > — "No HTTP transport"; `mcp serve` is spawned per-invocation over stdio). There is **no HTTP endpoint and
  > no port**, and `DashboardData` carries no base URL/port. A copy-paste `http://…/mcp` block would be
  > **fabricated functionality** (invariant #3). So the Connect-an-Agent block is a **`ComingSoon`** surface:
  > it shows the *intended* one-endpoint + key→profile model (visually, disabled, with the blue endpoint/
  > namespace styling) and a one-line hint that the working config today is stdio CLI, e.g. *"Today, agents
  > connect over stdio with `junction mcp serve --profile <name>`. A shared HTTP endpoint with keys is coming
  > soon."* The HTTP single-endpoint + junction-keys auth is its own later increment (§1 non-goals +
  > `docs/futures/revisit-when.md`). This keeps v1 truthful.
- **`routes/credentials.tsx` — FULL (wired).** Restyle to the new system. **Group by platform** so the
  multi-account wedge reads as multiple accounts under one source heading. Keep the inc-24 add/rotate/delete
  dialogs + `router.invalidate()` + toasts (re-skin them; do not change the mutation server-fns). Secrets
  never shown. Delete-while-routed already errors with the in-use message — surface it cleanly.
- **`routes/profiles.tsx` — FULL read.** Each profile a card: its **route rows** (read-only), a "N keys
  active" line → **ComingSoon**, and `New Profile`/`Add Route`/`Edit tool access` → **ComingSoon**. **No
  per-profile endpoint URL** (single-endpoint model) — drop `mcpEndpointPath` from the display (and from any
  `-profiles.test.tsx` assertion that checks it).
  > **Route-row field mapping (use the REAL `SourceMeta` shape — do not guess):** `SourceMeta = { namespace,
  > platform, credentialAccount, enabled, toolFilter?: { allow?: string[]; deny?: string[] } }` (from
  > `data.server.ts`). Render as `platform → credentialAccount → <namespace chip> · <filter> → <on/off>`.
  > Specifics: the account field is **`credentialAccount`** (not `account`); a **public/no-auth** source has
  > `credentialAccount === "(none)"` → render that as the **No auth** status (DESIGN.md status table), NOT a
  > literal "(none)" string; `toolFilter` is an **object** → render it compactly (e.g. `+2 allow` / `-1 deny`,
  > or "all tools" when absent), never `[object Object]`; `enabled` → the on/off dot+label.
- **`routes/platforms.tsx` — LIGHTER re-skin.** Ruled list: Name · Kind (mono tag) · Connections count ·
  Detail; row → simple detail (kind-specific descriptor read from `PlatformMeta` + which profiles route it,
  if derivable). `Add Platform` → **ComingSoon** (inc 25). Keep it lean.
