# Method File 04 — Data Model (Increment 4)

> **The conceptual spine.** Fills the `schema/` and `ids/` module seams in `@junction/core` with the entity schemas — `Platform`, `Credential`, `Profile`, `SourceRef` — as Zod schemas + inferred types + branded IDs, encoding the **multi-account wedge** and the load-bearing naming conventions. **Schema only — no persistence (that's increment 5), no CLI change.**
>
> **Builder:** Sonnet. These types become the public contract every later increment depends on — get them exactly right. The `ce-api-contract-reviewer` reviews this (exported types).

---

## Part 1 — Spec (what & why)

### Goal

Define junction's core entities as validated schemas, with the wedge and conventions baked in so renaming later can't happen (it would break every agent prompt). This realizes design spec §4. Everything is exercised by **tests** (the increment's proof) — there's no persistence or CLI yet, but the schemas, branded IDs, validators, and derivation helpers are all real, runtime-tested code.

### The entities (from design spec §4 — match exactly)

> **`Credential` — not `Platform` — is the unit a `Profile` references.** This single decision encodes the multi-account wedge.

- **`Platform`** — a supported external system, exists **once** in the catalog. `id`, `kind` (`mcp|openapi|graphql|cli|custom`), `displayName`, optional `specUrl`/`baseUrl`.
- **`Credential`** — one account's keys for one Platform. **MANY per Platform** (the wedge). `id`, `platformId` (→ Platform), `profileName` (`"work"`/`"personal"`), `kind` (`api-key|bearer|oauth2|file|env`), **`secretRef`** (see below — NOT the secret), optional `oauthMeta` slot.
- **`Profile`** — what an agent sees; gets its OWN MCP endpoint. `id`, `name`, `sources: SourceRef[]`, `mcpEndpointPath` (`/profiles/{name}/mcp`).
- **`SourceRef`** — an activated (Platform, Credential) pair inside a Profile. `platformId`, `credentialId`, `toolNamespace` (`"github_work"`, collision-free), `enabled`.

### Load-bearing conventions (encode as validators/helpers — renaming later breaks agent prompts)

- **`<namespace>__<tool>`** double-underscore tool namespacing. `toolNamespace` must match `^[a-z0-9_]+$` (lowercase, no `__` inside a namespace segment). A helper `namespacedTool(namespace, tool)` → `${namespace}__${tool}`.
- **Per-profile endpoint** `/profiles/{name}/mcp` derived from the profile name. A helper `deriveMcpEndpointPath(name)`; profile `name` validated (`^[a-z0-9-]+$` — URL-safe).

### Key design decisions (decided — implement exactly)

1. **`secretRef`, NOT `secret`.** Per `docs/rules/data.md` + `docs/rules/security.md` (secrets-as-references), the `Credential` schema holds an **opaque handle** (`secretRef: string`) to where the secret lives — the actual encrypted secret lives via the `CredentialStore` (increment 6). The persisted entity **never** contains plaintext or ciphertext. The schema must have NO field that could hold a raw secret value. Comment that increment 6 gives `secretRef` its meaning.
2. **`oauthMeta` reserved slot.** Optional, minimal (`{ scopes?: string[]; expiresAt?: string | null }` or similar) — present day one so the OAuth increment needs no migration (spec §4a). Keep it tiny; the OAuth increment fleshes it out.
3. **Branded IDs.** `PlatformId`, `CredentialId`, `ProfileId` as Zod-branded strings (`z.string().min(1).brand("PlatformId")`). Generated by ULID in `ids/`. **ID-format validation is deliberately light** (non-empty string) so the ULID→uuid swap stays a one-file change in `ids/` — `ids/` is the single generation swap point.
4. **The wedge is testable:** one `Platform{id: github}`, two `Credential`s both `platformId: github` with different `profileName` (`work`, `personal`) — both valid, distinct. This is the increment's headline proof.

### New deps (to `@junction/core`)

- `ulid@^3` (runtime). Zod already present (catalog).

### Module structure (per `docs/principles/modularity.md` — single purpose per file)

```
core/src/schema/
  index.ts        ← module barrel (curated re-exports of the schema public surface)
  primitives.ts   ← branded ID schemas (PlatformId/CredentialId/ProfileId), the
                    toolNamespace + profileName refinements, namespacedTool +
                    deriveMcpEndpointPath helpers
  platform.ts     ← PlatformSchema + Platform + PlatformKind
  credential.ts   ← CredentialSchema + Credential + CredentialKind (+ secretRef, oauthMeta)
  profile.ts      ← ProfileSchema + Profile
  source-ref.ts   ← SourceRefSchema + SourceRef
core/src/ids/
  index.ts        ← ULID generation: newPlatformId/newCredentialId/newProfileId
                    (the single ulid↔uuid swap point), branding generated IDs
```
`ids/` imports the brand types from `schema/primitives` (ids → schema, one direction, no cycle — schema never imports ids).

### Proof of done

- `pnpm verify` passes with **real behavior tests** (Vitest): the wedge (two creds, one platform); valid entities parse; invalid ones rejected (bad `toolNamespace`, bad `kind`, empty id, bad profile name); `namespacedTool`/`deriveMcpEndpointPath` produce correct values; ULID generators produce distinct, valid, correctly-branded IDs.
- **No field in any schema can hold a plaintext/ciphertext secret** — assert `secretRef` is a reference, and add a test/comment that the credential schema rejects/has-no `secret` field.
- The public schema surface is re-exported through `core/src/index.ts` (narrow barrel, named exports — NO `export *`).
- `pnpm depcruise` clean (intra-core only; `ids → schema` is fine, no cycle).
- SPDX headers; committed; PR green.

### Out of scope

- Persistence / Drizzle / DB (increment 5 — that's where `data.md`'s migration rules activate).
- The `CredentialStore` / actual secret encryption (increment 6 — inc 4 only defines `secretRef`).
- OAuth refresh logic (later — only the `oauthMeta` slot here).
- Any CLI change (`junction profile list` is increment 5, when there's something to list). MCP serving (increment 7).
- A platform catalog / seed data — inc 4 is the *shapes*, not instances.

## Part 2 — Implementation (step by step)

### Step 1 — `schema/primitives.ts`

- Branded ID schemas: `PlatformId`, `CredentialId`, `ProfileId` via `z.string().min(1).brand("…")`; export the inferred types (`type PlatformId = z.infer<typeof PlatformIdSchema>`).
- `ToolNamespaceSchema` = `z.string().regex(/^[a-z0-9_]+$/)`; `ProfileNameSchema` = `z.string().regex(/^[a-z0-9-]+$/)`.
- Helpers: `namespacedTool(namespace: string, tool: string): string` → `${namespace}__${tool}` (validate namespace via the schema first); `deriveMcpEndpointPath(profileName: string): string` → `/profiles/${profileName}/mcp` (validate name first).
- Keep these pure; no I/O.

### Step 2 — entity schemas (`platform.ts`, `credential.ts`, `profile.ts`, `source-ref.ts`)

Match design spec §4 exactly. Each file: the Zod schema + the `z.infer` type + any local enum (`PlatformKind`, `CredentialKind`). Use the branded ID schemas from `primitives`. Examples (adapt to Zod v4 idioms):
```ts
// credential.ts
export const CredentialKind = z.enum(["api-key", "bearer", "oauth2", "file", "env"])
export const OAuthMetaSchema = z.object({
  scopes: z.array(z.string()).optional(),
  expiresAt: z.string().nullable().optional(),
}) // reserved slot — OAuth increment fleshes this out
export const CredentialSchema = z.object({
  id: CredentialIdSchema,
  platformId: PlatformIdSchema,
  profileName: z.string().min(1),
  kind: CredentialKind,
  secretRef: z.string().min(1), // opaque handle → CredentialStore (increment 6). NEVER the secret.
  oauthMeta: OAuthMetaSchema.optional(),
})
export type Credential = z.infer<typeof CredentialSchema>
```
- `ProfileSchema`: `sources: z.array(SourceRefSchema)`, `mcpEndpointPath` validated (starts `/profiles/`, ends `/mcp`).
- `SourceRefSchema`: `toolNamespace: ToolNamespaceSchema`, `enabled: z.boolean()`.

### Step 3 — `schema/index.ts` (module barrel)

Curated named re-exports of the public schema surface (schemas + types + the helpers). No `export *`.

### Step 4 — `ids/index.ts`

- Lazy or direct import `ulid`. `newPlatformId(): PlatformId` = `PlatformIdSchema.parse(ulid())` (or cast — but parse keeps the brand honest). Same for `newCredentialId`, `newProfileId`.
- A comment: **this is the single ulid↔uuid swap point.** Nothing else generates IDs.

### Step 5 — wire the core barrel (`core/src/index.ts`)

Add the new public surface to the existing narrow barrel — the entity schemas + types + ID generators + the convention helpers. Named exports only. Keep `testing/` on its subpath.

### Step 6 — tests (colocated `*.test.ts`)

Behavior tests (no fs needed — pure schema, so no `JUNCTION_HOME` required, but follow Vitest conventions):
- **The wedge:** build one `Platform{id, kind:"mcp", ...}` and two `Credential`s with the SAME `platformId` and DIFFERENT `profileName` — both `.safeParse` succeed, IDs distinct. This is the headline test.
- **Valid parse:** a full `Profile` with a `SourceRef` parses.
- **Invalid rejects:** bad `toolNamespace` (`"Github Work"` — spaces/caps) rejected; bad `kind` rejected; empty `id` rejected; bad `profileName` (`"Work!"`) rejected; `mcpEndpointPath` not matching the pattern rejected.
- **Helpers:** `namespacedTool("github_work", "list_issues")` === `"github_work__list_issues"`; `deriveMcpEndpointPath("work")` === `"/profiles/work/mcp"`; both reject invalid input.
- **IDs:** `newPlatformId()` is distinct across calls, non-empty, parses as `PlatformIdSchema`.
- **Security:** assert the `CredentialSchema` has `secretRef` and that an object carrying a literal `secret` field doesn't smuggle it through (strip/ignore via Zod's default object behavior — confirm no `secret` survives parse).

### Step 7 — deps, verify, commit

- `pnpm add --filter @junction/core ulid@^3`.
- `pnpm install`; `pnpm verify`; `pnpm build`; `pnpm depcruise` (clean — confirm `ids→schema` intra-core, no cycle, no boundary violation).
- SPDX headers on all new `.ts`. Commit; push branch; open PR (base main): "feat: data model — Platform/Credential/Profile/SourceRef (increment 4)".

## Review (background, after build)

- Junction: `junction-package-boundary` (intra-core `ids→schema`, no cycle), `junction-clean-code-reviewer` (no `secret` field — secrets-as-references; narrow barrel; single-purpose files; the `__`/endpoint conventions encoded; derive types from one source).
- CE: **`ce-api-contract-reviewer`** (the exported type contract — these become load-bearing public types; naming, optionality, evolvability), `ce-correctness-reviewer` (the wedge actually works; validators catch the right things; branded IDs don't leak), `ce-testing-reviewer` (valid + invalid coverage, the wedge test, helper edge cases), `ce-security-reviewer` (the secrets-as-references invariant: no schema field can hold plaintext).
- Then `/ce-simplify-code` on the diff.
