## Part 1 — Spec (what & why)

### Goal

Add the standard OSS community files so contributors (human and agent) know how to engage, and so the repo signals quality: README, CONTRIBUTING, CODE_OF_CONDUCT, SECURITY (credential-handling disclosure), issue/PR templates, and the SPDX-header policy decision. AGPL means contribution terms and the §13 network-source-offer get a real mention.

### Deliverables

| # | File | Purpose |
|---|---|---|
| 1 | `README.md` | What junction is, the wedge, status (foundation in progress), quick links, license. Replace the current stub. |
| 2 | `CONTRIBUTING.md` | How to contribute: the method-file workflow, `pnpm verify` gate, `pnpm changeset` for user-facing changes, AGPL contribution note, the rules/principles pointers. |
| 3 | `CODE_OF_CONDUCT.md` | Contributor Covenant 2.1, with a real contact. |
| 4 | `SECURITY.md` | Disclosure policy — **especially important: junction handles credentials.** Private reporting (GitHub Security Advisories), scope, response expectations, what NOT to put in public issues. |
| 5 | `.github/ISSUE_TEMPLATE/bug_report.yml` + `feature_request.yml` + `config.yml` | Structured issues; config points security reports to SECURITY.md, not public issues. |
| 6 | `.github/PULL_REQUEST_TEMPLATE.md` | Checklist: tests pass (`pnpm verify`), changeset added if user-facing, rules/principles followed. |
| 7 | `docs/rules/licensing.md` (or a section) | The SPDX-header policy (`AGPL-3.0-only`, one-line headers, enforcement deferred) + the AGPL §13 source-offer product requirement. |

## Part 2 — Implementation

### Step 1 — `README.md`

Replace the stub. Sections (keep tight — this is a foundation, don't oversell):
- **One-liner + the wedge:** self-hosted, single-user broker; the one place you connect platform accounts once so any AI agent reaches them over MCP/CLI/API; the multiple-accounts-per-platform angle. Link `docs/idea.md`.
- **Status:** "Foundation in active development — not yet usable. See `docs/specs/` and `docs/methods/` for the build plan." Be honest; no fake install instructions.
- **Architecture (brief):** the package shape + "core is pure, edges thin." Link the design spec.
- **Contributing / License:** link CONTRIBUTING; "AGPL-3.0 — see LICENSE." Note the §13 network-use implication in one line.
- No install/usage section until there's something to install (avoid lying docs).

### Step 2 — `CONTRIBUTING.md`

- The **method-file workflow** (`docs/workflow.md`): research → method file → approve → build → review → test. Link it.
- **Gate:** every change passes `pnpm verify`; hooks enforce it. Link `docs/rules/` + `docs/principles/`.
- **Changesets:** "user-facing change → run `pnpm changeset`."
- **Commits/PRs:** PR title convention (only if 0.75 added PR-title lint); otherwise just "clear PR description." Note commit-level linting is intentionally not required.
- **AGPL contribution note:** contributions are licensed under AGPL-3.0-only; by contributing you agree to that. (No CLA unless the user wants one — default: no CLA, inbound=outbound AGPL.)
- **Dev setup:** Node 22 (floor 20), pnpm, `pnpm install`, `pnpm verify`. Link `junction-dev` skill content / `docs/`.

### Step 3 — `CODE_OF_CONDUCT.md`

Contributor Covenant 2.1 verbatim, with the enforcement contact = the user's email (`hey@amitray.dev`) or a project address the user prefers — **ask which** before hardcoding a personal email publicly; default to a placeholder `conduct@…` note if unsure.

### Step 4 — `SECURITY.md` (credential-handling — get this right)

- **Reporting:** use **GitHub Private Security Advisories** (link), not public issues. Provide a fallback contact.
- **Critical for junction:** explicit instruction — *never paste real credentials, tokens, or secrets into issues, PRs, or advisories; redact them.*
- **Scope:** what's in scope (the broker, credential layer, sandbox, MCP endpoints) and out (third-party MCP servers junction connects to).
- **Response expectations:** acknowledge within N days, coordinated disclosure.
- **Supported versions:** "pre-1.0, only `main` is supported" for now.

### Step 5 — Issue + PR templates

- `.github/ISSUE_TEMPLATE/bug_report.yml` (YAML forms): repro, expected/actual, version/commit, environment.
- `.github/ISSUE_TEMPLATE/feature_request.yml`: problem, proposed solution, alternatives.
- `.github/ISSUE_TEMPLATE/config.yml`: `blank_issues_enabled: false`; a contact link routing **security → SECURITY.md** (so vulns never land in public issues).
- `.github/PULL_REQUEST_TEMPLATE.md`: checklist — `pnpm verify` passes; changeset added if user-facing; follows `docs/rules/` + `docs/principles/`; tests included.

### Step 6 — `docs/rules/licensing.md`

- **SPDX policy:** every source file *will* carry `// SPDX-License-Identifier: AGPL-3.0-only` (one-line, REUSE-style). **Enforcement deferred** to the security increment via `fsfe/reuse-action`; insertion via tooling, not by hand. State `AGPL-3.0-only` (specific version, not `-or-later`).
- **AGPL §13 product requirement:** the running broker must offer its Corresponding Source to network users — embed git commit/version at build time and surface a source link (in `--version`/about, MCP server-info, or a startup log). This is a **product requirement** for the broker increment, recorded now so it isn't forgotten.
- Cross-link from `docs/rules/security.md` and `docs/rules/README.md`.

### Step 7 — Verify & commit

- Confirm `config.yml` disables blank issues + routes security correctly.
- `pnpm verify` green.
- `git add -A && git commit`.
