## 1. The locked domain model (confirmed complete by the user)

Junction is a single-user patch bay: you wire accounts (connections) into bundles (profiles); each
bundle is reachable by an agent. The intricacy concentrates in one object, the **source_ref** (the
routing cable). Verified against the schema.

**Entities + cardinalities**
- **Source** (Platform): has a KIND (`mcp | openapi | graphql | cli | custom`); kind-specific descriptor
  (mcp → url/command, openapi → spec + operations, graphql → endpoint, cli → sandboxed command).
- **Connection** (Credential): one account on ONE source ("work"); a secret; OR a source has ZERO
  connections (public/no-auth). MANY per source = the **multi-account wedge**.
- **source_ref** (the routing cable): `(profileId, platformId, credentialId?, toolNamespace, enabled,
  toolFilter?)`. `credentialId` is **nullable** (public) and **can repeat** across rows/profiles.
  `uniqueIndex(profileId, toolNamespace)`.
- **Profile**: a set of source_refs; today 1:1 with a `/profiles/{name}/mcp` path. **(inc 24.5 reframes
  this to a single shared endpoint — see §3.)**
- **Agent / junction key** (future): a key scoped to a profile (or global / multi-profile); many per profile.

**The 7 edge cases the UI must be able to express** (the route row makes each legible):
1. **One connection reused across profiles** — same credential, DIFFERENT namespace + filter per profile.
2. **Same source, different accounts in different profiles** (wedge splits).
3. **Source with NO connection** (public) → badge "No auth", credential optional.
4. **namespace UNIQUE within a profile** → same source twice needs distinct ns (gh, gh2); UI enforces/suggests.
5. **Per-routing toggles** — `enabled` + `toolFilter` live on the source_ref (on in one profile, off/filtered in another).
6. **Delete rules** — delete profile → source_refs cascade; delete a connection still routed → BLOCKED
   (RESTRICT) naming the profiles; rotate secret → all profiles keep working.
7. **Consumption fan-out** — one endpoint, many agents/keys; revoke one without the others; per-key usage.

Real data shapes the web reads: `DashboardData`, `PlatformMeta`, `CredentialMeta`, `ProfileMeta` +
`SourceMeta` (see `packages/web/src/server/data.server.ts`). Only credential add/rotate/delete mutate (inc 24).

## 2. Why the design rewrite (and what failed first)

The inc-23 "instrument" system (amber/zinc/Departure-Mono, the left-edge live rail, mono eyebrows
everywhere, 13px body) read **slop-adjacent**. Several prototype directions (a dense instrument, a
warm-paper editorial "Ledger", a soft-product lilac take) were built and rejected: the lesson, from
`docs/design/anti-ai-slop.md` + impeccable.style/slop, is that **slop is structural, not cosmetic** —
painting onto a generic sidebar + card-list chassis and decorating it. The non-slop win comes from a
structure specific to what junction *is* (routing), plus restraint.

Tells the earlier attempts tripped (now forbidden, see `anti-ai-slop.md`): side-accent nav bar (the #1
tell), hero-metric stat banner, identical card grids, mono-eyebrows-everywhere, dark-mode-with-glow,
cream/beige page, 13px body, italic-serif hero, em-dash overuse, "theater" framing.

## 3. The Geist study (the chosen bar) + the two product directives

The user set the quality bar at Vercel **Geist** (vercel.com/design.md, /design/guidelines, dark tokens,
/geist/introduction). Adopted into `DESIGN.md`: numbered-intent color tokens (gray ranks info; one accent
for state + the single primary action), Geist type tokens (heading/label/copy/button + mono; copy-14/
label-14 default), 4px spacing with the 8/16/40 rhythm, 6/12 radii, **subtle two-layer shadows** (reverses
inc-23's borders-only), 0/150/200/300ms motion never `transition:all`, and the Geist copy voice.

Two resolutions:
- **Geist Sans/Mono is now a documented choice** (the font is on the "overused" slop list, but the user's
  bar is explicitly Geist → deliberate, logged in DESIGN.md).
- **Blue is the accent** (Geist's signature; the generic "no blue" rule is overridden for this project, logged).

**Two product directives from the user:**
1. **Dark + light, both first-class** (Geist has both via same token names).
2. **"Profiles shouldn't have their own endpoints"** → the UI adopts a **single shared MCP endpoint**;
   a junction key selects which profile(s) an agent gets. v1 *shows* this model (no per-profile URLs);
   the key-auth **backend** is a later increment. ⚠️ This intentionally **revises the load-bearing
   CLAUDE.md invariant** "Per-profile MCP endpoints, not shared-endpoint filters" — when the auth backend
   is built, update CLAUDE.md + the foundation design spec. Recorded in `docs/futures/revisit-when.md`.

## 4. Scope decisions (user, 2026-06-29)

- **inc 24.5 = the setup / foundation increment** ("split 24.5 to just the setup idea; inc 25 is the real work").
  It rewrites the design system + rebuilds the 4 surfaces; it does NOT add new functional backends.
- **Build depth:** Credentials FULL (wired, inc-24 ops) · Dashboard FULL (single-endpoint Connect-an-Agent =
  the main new value) · Profiles FULL read (route rows) · Platforms LIGHTER re-skin.
- **"Coming soon" affordance** on everything deferred (recent activity → inc 29; Add Platform → inc 25; New
  Profile / Add Route / Edit tool access → inc 26; Manage Keys / single-endpoint auth → keys increment),
  with a CLI hint. v1 stays truthful: only credential mutations actually run.
- The ambitious "grand reframe" (connections vocabulary, the deeper consumption/agents surface) remains a
  pre-public-release future; only the *presentation* evolves later, the model in §1 is the durable truth.

## 5. Prior-art reference (connect-once UX, for later increments)

From the Composio/Zapier/Pipedream/n8n/Make/Raycast research (full report in git history of the planning
session): the connect-once catalog pattern, modal connect flow (OAuth + API-key dual path), multi-account
"select existing or add", "connection" as the user-facing noun for a credential, progressive disclosure
(catalog card → detail → advanced), and the SaaS-only patterns to DROP (billing, teams, community tabs,
usage metering per connection). These inform inc 25+ (platform/connect flows) and the eventual reframe,
not inc 24.5.
