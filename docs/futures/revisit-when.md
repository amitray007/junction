# Revisit-when — deliberately deferred decisions

Things we chose **not** to build/adopt yet, each with the **trigger** that should make us reconsider. Deferral is a feature: we keep the foundation lean and add weight only when a real need arrives. When a trigger fires, revisit the row and record the increment that resolved it.

| Deferred | Trigger to revisit | Notes |
|---|---|---|
| **Effect-TS** (structured concurrency / typed effects) | Concurrent multi-source MCP fan-out, a sandbox process-pool, or cancellation-heavy flows make neverthrow's manual `Result` composition genuinely painful | Today: **neverthrow** is the error model (spec §5a). The exception is narrow — only if structured concurrency earns its keep. Do not adopt for error-handling alone. |
| **Warm-pool / persistent upstream sessions** (`UpstreamSession` reuse) | Per-request connect latency becomes measurable (> ~50 ms p50 on a real workload) or a benchmark shows pool savings outweigh complexity | Today: `createProfileProxy` creates a fresh session per `listTools`/`callTool` call and closes it immediately after (connect-per-call v1, inc 12). Warm-pool would cache a live `UpstreamSession` per (platformId, credentialId) pair and health-check it on reuse. (inc 11, 12) |
| **Sandbox for third-party stdio MCP binaries** | Junction spawns an untrusted or third-party MCP binary via `StdioClientTransport` | Today: the stdio transport spawns the binary without OS-level sandboxing (first-party / trusted sources only). When untrusted binaries are supported, route the spawn through the inc-8 sandbox (`createSandbox`) so the child runs under Seatbelt / bubblewrap. (inc 11) |
| **Long-running daemon** (`junction serve` as a persistent process) | MCP-serving needs a shared long-lived process, or multiple concurrent agents, or Streamable HTTP | Today: `mcp serve` is a per-invocation stdio process (inc 7). `core` stays daemon-free and embeddable. |
| **Streamable HTTP transport** (MCP) | Remote / networked MCP serving arrives | Today: **stdio only** (inc 7). SSE is already deprecated and avoided. HTTP brings Origin / DNS-rebinding concerns — out of scope until then. |
| **better-auth** | Remote / multi-device **web login** is needed | Human web login only; **never** owns platform tokens (those live in the encrypted vault in `core`). |
| **arctic OAuth vault + refresh loop** | Real **OAuth platform sources** connect | The credential store (inc 6) already holds encrypted secrets; arctic adds the OAuth dance + token refresh at that increment. |
| **microsandbox / libkrun microVM** (escalation tier) | Running **hostile code / arbitrary npm**, or needing stronger-than-OS-sandbox isolation | Also the **forward path** for the Seatbelt deprecation (see `deprecations.md`). Drops in behind the existing `Sandbox` interface. |
| **tsgo** (TS7 native compiler) | GA + stable | Swap the `tsc -b` typecheck binary; no code change expected. |
| **Valibot at the web edge** | Web **bundle size** becomes a *measured* constraint | Zod v4 stays in `core` (the boundary validator). Don't pre-optimize. |
| **libsql** (swap from better-sqlite3) | **Whole-DB at-rest encryption** is wanted | Secrets-as-refs (inc 5/6) already neutralizes a DB leak; libsql only if full-DB encryption is independently desired. |
| **pino structured audit log** | An **audit trail** of tool calls / credential use is needed | Recorded in the stack; installed at its increment. |
| **AGPL §13 network-source-offer** | The broker **serves network users** | Must surface a source link + embed the running commit/version so network users can obtain source. Wire at the broker/serving increment. |
| **SPDX header CI enforcement** (`fsfe/reuse-action`) | Security-hardening increment | Policy is in force now (every file carries SPDX); the *CI gate* lands with the other security tooling. |
| **Deferred CI tooling** — knip (dead code/deps), type-coverage ≥99%, publint+attw, targeted semgrep (sandbox/secrets), CodeQL, osv-scanner, trufflehog, ruleset-drift check, signed commits | Each at the increment that needs it (mostly the security increment; publint/attw when a package first publishes) | Spec §5b. Wired incrementally, not upfront. |
| ~~**react-doctor**~~ → **ADOPTED inc 23** (on-demand, not in-loop) | ~~trigger fired~~ | Resolved inc 23: adopted as the on-demand React audit (`npx react-doctor@latest --verbose packages/web`), run at the web-increment review step — NOT wired into `pnpm verify`/per-edit hooks (spec §5b). Usage + triage convention in `docs/rules/web.md` (react-doctor section). Baseline taken inc 23 (44 raw diagnostics; true positives fixed in the anti-slop pass). |
| **eslint-plugin-react-hooks** | ESLint is adopted as the project linter (replacing Biome) or Biome drops `useExhaustiveDependencies`/`useHookAtTopLevel` | Biome covers hooks linting (useExhaustiveDependencies, useHookAtTopLevel). ESLint-as-the-loop is banned; adding eslint-plugin-react-hooks would require ESLint. Revisit only if Biome's hooks rules become insufficient. (inc 22) |
| **Tailwind CSS** | The web UI grows beyond the minimal read-only dashboard (mutations, forms, richer data visualization) | inc 22 uses a single `app.css` (~100 lines). Tailwind v4 (CSS-native, no PostCSS dependency) is the preferred path. (inc 22) |
| **Browser e2e tests (Playwright)** | The web UI has mutations or complex interactive flows that need regression coverage | inc 22 is read-only; unit tests cover server data fns. Add Playwright when forms/mutations land in inc 23+. (inc 22) |
| **Changesets publishing** | A package is actually **published** | Currently disabled (`private: true` + `changeset tag`); versioning works, publishing is intentionally off. |
| **Credential key rotation** | Operator needs to rotate the master key | The encrypted store supports re-encrypt in principle; no `rotate` command yet. |
| **OpenTUI (Node FFI)** | OpenTUI ships a Node-compatible native renderer | Increment 9 uses **Ink** for the TUI dashboard. OpenTUI's `bun:ffi`-backed renderer throws `"native FFI is not available for this runtime yet"` under Node. When OpenTUI ships a Node-safe renderer, consider migrating the TUI from Ink. See `docs/futures/gotchas.md` (OpenTUI) for the empirical finding. |
| **Live config reload** (source toggles / add / remove take effect on next `mcp serve`) | Mid-session live updates are wanted (e.g. disable a source without reconnecting) | Currently `createProfileProxy` snapshots `enabledSources` once at server construction — a toggle or remove persists to the DB but only takes effect on the next `mcp serve` invocation. MCP clients spawn a fresh serve per session so this is acceptable for v1. When live updates are needed, revisit: re-read `profile.sources` on each `listTools` call, or add a reload signal. (inc 13) |
| **Egress sandboxing for untrusted OpenAPI hosts** | Junction calls arbitrary user-supplied `baseUrl` targets | Currently `callOperation` validates the base URL is `https?://` and pin-checks it against the `connection.baseUrl` / `spec.servers[0].url`, but makes no OS-level egress restriction. When untrusted or multi-tenant specs are admitted, route HTTP calls through the inc-8 sandbox to block SSRF to internal hosts. (inc 15) |
| **Spec complexity guard at add-time** | A spec with a billion-laughs / deep-fan-out `$ref` structure causes `dereference()` to produce a huge in-memory object and an equally huge cache file | Currently `parseSpec` has a 10 MB byte cap on the raw spec download and a 30 s timeout on fetch, but no guard on the dereferenced output size. When spec caching grows or complex nested-$ref specs become common, add an add-time complexity check (e.g. cap the serialised `JSON.stringify(schema)` length or the node-count from `dereference`). Trigger: dereferenced cache files regularly exceed 50 MB or user reports OOM during `platform add`. (raised inc 15) |
| **Full AJV validation of request args against the operation schema** | Agents send systematically bad input (wrong types, missing required fields) and users want structured validation errors instead of upstream 4xx | Currently `callOperation` does light path-injection guards + 4xx passthrough. AJV (or `@cfworkers/ajv`) wired to the `inputSchema` would catch malformed calls before the network hit and return a typed `invalid-args` result. Add at the inc that introduces agent-facing CLI tool-call UX. (inc 15) |
| **Rich platform selection for OpenAPI** (`platform add --kind openapi` selector UX) | More than a handful of OpenAPI sources are being managed and users want a TUI browser | Currently the CLI takes all flags upfront; no interactive tag/operation picker. Wire as a follow-on to the TUI increment (inc 16+). |
| **GraphQL query cost / depth limiting** (`graphql_query`, `graphql_mutation`) | A GraphQL source is connected to an API that has no native cost limiting and agents send complex nested queries causing server-side DoS, or rate-limiting bites | Currently `callGraphQl` enforces `maxQueryBytes` (byte-level guard) but has no per-field depth or complexity budget. When needed, add `graphql-depth-limit` or `graphql-query-complexity` evaluation before `callGraphQl`; gate on a `maxDepth` / `maxComplexity` field in `GraphQlConnection`. (inc 20) |
| **Per-field typed tools for GraphQL** (replace `graphql_query` + `graphql_mutation` with one tool per SDL operation) | Agents reliably have the SDL cached and the MCP client can benefit from a richer, per-operation input schema | Currently the provider exposes 3 fixed tools (query, mutation, schema). When SDL-awareness matures and per-operation schemas add enough value to justify the increased tool count, generate one `Tool` per top-level field on `Query` / `Mutation`. (inc 20) |
| **Extract GraphQL + OpenAPI HTTP executor utilities to `@junction/core`** | A 3rd consumer of the streaming body-read loop, byte-cap logic, or AbortError catch block appears | Currently duplicated (with `jscpd:ignore` markers) between `openapi-client/http.ts` and `graphql-client/http.ts` — deliberate 2nd-use hold until the rule of three fires. When a 3rd package needs these, extract `readBodyCapped`, `isAbortError`, and a shared `callFetch` helper into core. (inc 20) |
| **`credentialEnvVar` denylist refinement** — tighten or expand the `/_TOKEN$|_SECRET$|_KEY$/` schema guard | `validatePolicy`'s denylist evolves (e.g. adds `_PASSWORD` or `_CREDS`) AND `CliConnectionSchema` doesn't mirror it → a name that passes schema validation at `platform add` still gets rejected at run time | Currently the schema `.refine` and `validatePolicy`'s `SECRET_DENYLIST_RE` are intentionally in sync (both reject `_TOKEN/_SECRET/_KEY`). If one changes without the other, users see a confusing "schema-valid but runtime-rejected" error. Keep the two lists in lock-step; they live in `cli-connection.ts` and `sandbox.ts` respectively. (inc 21) |
| **OS-level egress control for the CLI command tier** | Untrusted operators, multi-tenant deployments, or a specific compliance requirement makes uncontrolled network egress by CLI commands a risk | Currently `CliPolicy.allowNet` is declared in the descriptor and validated by `validatePolicy`, but is NOT enforced by the seatbelt/bubblewrap harness for `runCommand` (only `runScript`/Deno enforces `--allow-net`). A CLI command can reach any host unless the host itself firewalls it. Trigger: when host-level firewall is insufficient and per-command egress control is required. Forward path: map `allowNet` → `(deny network*)` + per-entry `(allow network* (remote …))` in the Seatbelt profile; bwrap equivalent. (inc 21) |
| **Code-mode execution layer (QuickJS-WASM over the `ToolProvider` proxy)** | The base (sources + credentials/OAuth + serving) is solid and agents need to *orchestrate* many tool calls cheaply (write code instead of N round-trips) | The planned **fast** execution path, distinct from the OS sandbox (which is for real *local binaries*). The agent writes JS that calls a `tools.*` proxy; the snippet runs **in-process in a QuickJS-WASM isolate** (capability-secure — no ambient FS/net, can only call the proxy; sidesteps the `node:vm`/isolated-vm ban) and each `tools.*` call dispatches to the existing core proxy → providers do the real I/O with injected creds. ~sub-ms/call vs ~25 ms for a sandboxed process. Reuses namespacing + credential injection + toolFilter we already have. Prior art: RhysSullivan/executor (`runtime-quickjs`, `runtime-deno-subprocess`). Build AFTER Web UI → OAuth → Audit. |
| **Per-profile HOME/config isolation for the `cli` source + env-vs-file credential split** | The same CLI tool is used across multiple profiles and must use each profile's OWN credentials/config, not the host's ambient ones | Per-profile credential isolation already comes from the **per-call fresh process + explicit scrubbed env**; the sandbox's FS read-confinement (inc 21) additionally stops file-credential tools (`aws`/`gh`/`git`/`kubectl`/`docker`) from falling back to the host's `~/.aws`/`~/.config` etc. **Two refinements when this is exercised:** (1) auto-provision a per-`(profile,credential)` HOME/config dir (writePath + `HOME=`/`AWS_CONFIG_FILE=` in env) so file-cred tools get profile-scoped state; (2) **env-cred tools** (support `GH_TOKEN`/`AWS_ACCESS_KEY_ID` etc.) need only fresh-process + scrubbed-env (~5 ms) — skip the ~25 ms FS-confining sandbox; reserve the sandbox for tools that insist on credential *files*. |
| **Seatbelt/bwrap per-call overhead (~25 ms) → warm-pool or light isolation mode** | `cli`-source tools are called at high frequency / on-the-fly and the ~25 ms/call spawn+profile-setup tax (measured: raw ~5 ms, sandbox-exec ~21 ms, + junction profile-temp-file) becomes a measured latency problem | The OS sandbox is inherently spawn-heavy; there is no "faster OS jail" (microVMs are *slower* to start). Levers: (a) the fast path for frequent work is **code-mode (in-process, no spawn)** + the HTTP providers (no sandbox) — don't route quick/API actions through the OS sandbox at all; (b) a **light isolation mode** (fresh process + scrubbed env, no Seatbelt) for *trusted env-cred* tools; (c) **warm-pool** persistent sandboxed processes (see the warm-pool row). Do NOT gold-plate the OS sandbox speculatively — it's the niche local-binary path. |
| **Normalize the migration journal (0003 timestamp) + add an old-DB-upgrade / monotonicity regression test** | BEFORE the distribution increment (publishing junction so real users exist), or if any future migration is hand-authored | `0003_add_openapi_column`'s journal `when` (`1782600000000`) is non-monotonic (> 0004/0005/0006), poisoning the drizzle better-sqlite3 high-water mark so later migrations are silently skipped on a DB created in the inc 15–20 window (see gotchas.md). **Today: low-impact** — fresh installs apply all from scratch and future migrations (real, larger timestamps) apply fine; only stale dev DBs are affected (no distributed users). **Before distribution:** (a) lower 0003's journal `when` to between 0002 (`1782426252591`) and 0004 (`1782483347172`) so the chain is monotonic for all new installs; (b) add a regression test — assert `_journal.json` `when` strictly increases by idx, AND seed a raw DB at an earlier migration state, run `getDatabase`/migrate, assert the `graphql`+`cli` columns appear (every existing test used a FRESH DB, which never trips the high-water — that's the gap that hid this). Note: editing the journal does NOT auto-repair already-stuck DBs (their stored `created_at` is unchanged) — fine while there are no real users; if real users ever predate the fix, a one-time `created_at` repair is needed. (raised inc 22) |
| **StatusRail live pulse** (amber flash on active tool call, per-segment activity) | MCP serve is wired to the web process AND a real-time channel (SSE or WebSocket) from inc 26+ | Currently `StatusRail` (`src/ui/status-rail.tsx`) renders static 4-px coloured segments from hardcoded placeholder data. The inc 23 rule: no live data until the real-time channel exists — polling or long-SSE is the trigger. (raised inc 23) |
| **Patch-bay / source-toggle UI** (enable/disable sources from the dashboard) | Write paths (mutations) land in the web UI (inc 26+) | The Profiles page shows read-only source status. Enabling/disabling a source is a write path; the web UI is currently read-only. Wire when mutations are added. (raised inc 23) |
| **Browser e2e tests (Playwright)** — full flow for route navigation + 404 | The web UI has mutations or interactive flows that unit tests can't cover | Currently unit tests cover server functions and component render; route navigation is tested by TanStack's own router tests. Playwright lands when write paths (forms, mutations) appear. (raised inc 22, reaffirmed inc 23) |
| **serve.mjs static-serve regression tests** (traversal guard + leak-grep negative path) | Next time `serve.mjs` or the CI leak-grep is edited, or when the first web test harness can drive the built server | The `resolveStaticFile` path-traversal guard (`startsWith(CLIENT_DIR + sep)`, blocks `../` + sibling-prefix `dist/client-evil`) is verified by manual fuzzing (15 vectors) but has **no automated test** — a future edit could weaken it silently.
