# Gotchas — fragilities already hit, worked around, worth remembering

Non-obvious sharp edges we've already paid for. Each lists the **symptom** and the **fix in place**, so the next person (or agent) doesn't re-learn it the hard way. Roughly grouped by area.

## Web mutations (inc 24)

- **The web write-path pattern (reuse it for inc 25–27).** A mutation is a **POST `createServerFn`** in a `*.functions.ts` module (the RPC boundary, client-importable) whose `.validator()` validates the input (server-side, pure — no core import, no state) and whose `.handler()` calls `assertLocalHost()` **then** a thin `*.server.ts` helper that imports `@junction/core` and does the work. The route calls the fn, then `await router.invalidate()` (the source of truth for refresh — the inc-23 `staleTime` makes nav cached, invalidate forces a fresh loader) + a sonner `toast`. Return **metadata-only** shapes, never raw core types, never the secret/secretRef. Memoized DB lives in `shared.server.ts` (`getDb()`), shared by data + mutations — never call `getDatabase()` per request (inc-22 connection-leak gotcha). (inc 24)

- **`assertLocalHost()` runs *inside* `.handler()`, which fires AFTER `.validator()`.** This is safe ONLY because the validators are **pure** (typeof+trim string checks, throw a 400 Response, touch no state/core/log). If a validator ever did real work, it would run before the loopback guard. Keep validators pure; the guard order is fine as long as they are. (inc 24, security review)

- **`input.secret = ""` after use is best-effort hygiene, NOT a memory wipe.** JS strings are immutable and not zeroable — the original plaintext stays in the V8 heap until GC, and reassigning the param can't affect the caller's copy. The *real* guarantee (secret never enters a return value, error, log, or response) is upheld structurally and tested; do not read the `= ""` as a security control. (inc 24, security + correctness review)

- **Rotation must stay in a neverthrow `.andThen` chain — do NOT use `ResultAsync.fromPromise(async…throw…, e => e as CredentialError|DbError)`.** The `as` cast is unsound: a raw rejection (contract violation) would be miscast, `.kind` becomes `undefined`, and the toast renders "…: undefined". `rotateCredential` mirrors `add-credential.ts`/`remove-credential.ts` exactly (pure chain, typed errors, no cast). Atomicity: new store write (fresh ULID secretRef) → DB repoint → on DB-fail clean up the new store entry (best-effort swallow) keeping the old ref live → on success delete the old store entry (best-effort). Fresh ULID per rotation also avoids AES-256-GCM IV/AAD reuse. (inc 24, correctness/credential review)

## Web design system (inc 23)

- **Tailwind v4 + TanStack Start plugin order is load-bearing.** The Vite plugin order in `packages/web/vite.config.ts` MUST be `tanstackStart()` → `tailwindcss()` → `viteReact()`. Swapping Tailwind before TanStack Start breaks SSR asset injection; swapping Vite React before Tailwind breaks class generation in JSX. The `@tailwindcss/vite` plugin (v4's Vite integration, replacing PostCSS) must be in the middle slot. If you see missing CSS in prod or a cold-start flash of unstyled content, check this order first. (raised inc 23)

- **Biome CSS parser requires `tailwindDirectives: true` to parse `@theme inline`.** Biome 2.5 does not recognise Tailwind v4's `@theme` at-rule by default — the CSS file will fail parsing with a cryptic "unexpected token" error. Fix: `"css": { "parser": { "tailwindDirectives": true } }` in `biome.json`. Additionally, Biome's CSS linter produces false positives on Tailwind utility classes injected by `@import "tailwindcss"` — disable it for CSS files via `overrides`: `{ "includes": ["**/*.css"], "css": { "linter": { "enabled": false }, "formatter": { "enabled": false } } }`. If Biome is upgraded and CSS formatting becomes desirable, remove the formatter override and test for regressions. (raised inc 23)

- **Client-bundle leak grep: code-splitting means one module can contaminate many chunks.** The leak grep (`grep -rl "better-sqlite3\|…" packages/web/dist/client`) catches the identifier anywhere in the client chunk set — it doesn't need to name the specific entry point. However, if a `createServerFn` boundary is accidentally removed or the module is imported via a dynamic `import()` without a server-only path guard, the native identifier will appear in a split chunk, not just the entry. When the grep fires, look at ALL matching files to trace the import chain. The `ssr.external` setting in `vite.config.ts` is defence-in-depth but cannot always prevent a leaked identifier from appearing as a string literal; the grep is the definitive gate. (raised inc 23)

- **Departure Mono is NOT on npm — it must be vendored from GitHub releases.** The OFL font (`DepartureMono-Regular.woff2`) is available only at the GitHub release (`github.com/renebigler/departure-mono/releases`). Download the zip, extract the woff2, and copy to `packages/web/src/styles/fonts/`. The `DepartureMono-OFL.txt` licence file MUST be committed alongside it. If the font is missing at build time, the `@font-face` declaration will silently fail (no console error, no flash — text just renders in the fallback monospace). Verify by inspecting the loaded fonts in DevTools: `document.fonts.check("12px 'Departure Mono'")` should be `true`. (raised inc 23)

## Web shell (inc 22)

- **`verbatimModuleSyntax: true` in tsconfig causes server bundles to leak into the client bundle** in TanStack Start. The base `tsconfig.base.json` has `verbatimModuleSyntax: true`, so `packages/web/tsconfig.json` does NOT extend it — it is a standalone tsconfig (no `extends`, no `composite`, `moduleResolution: bundler`). If you accidentally add `verbatimModuleSyntax: true` or extend `tsconfig.base.json`, native deps (`better-sqlite3`, `@napi-rs/keyring`) will appear in `.output/public` client JS. The client-bundle leak grep is the regression guard.

- **TanStack Start build output is a Fetch handler, not a standalone HTTP server.** `@tanstack/react-start` v1.168.x with Vite 8 SSR environments produces `dist/server/server.js` — a WinterCG-compatible Fetch handler (`server_default.fetch(req)`), not a runnable Node.js server. The `.output/server/index.mjs` path applies only when you add the `nitro()` Vite plugin from `nitro@nitro-nightly` (which was excluded — beta). Instead, `packages/web/serve.mjs` (a committed source file, not a build artifact) bridges this: it wraps the Fetch handler in a `node:http` server. `exports: {"./server": "./serve.mjs"}` and `scripts.start` both point at `serve.mjs`. The client-bundle leak grep target is `packages/web/dist/client` (not `.output/public`). If a future Start version produces a self-hosting server entry, replace `serve.mjs` and update `exports`. (inc 22)

- **`@junction/web` in cli `dependencies` is an ARTIFACT dep, not a code edge.** `junction web` resolves the built server path via `import.meta.resolve("@junction/web/server")` at runtime — NOT via a static `import`. The package.json dep enables the resolver to find `node_modules/@junction/web`; it creates no code import edge. depcruise sees no `cli→web` edge. If you see a depcruise `apps-dont-import-apps` violation after touching `web.ts`, check that you have not accidentally added a static `import from "@junction/web"` in the source (the resolve call is fine).

- **Server-only core boundary (triple guard):** `@junction/core` (and its native modules) must NEVER reach the client bundle. Three layers: (1) `createServerFn` strips the handler body from the client build — anything imported inside the handler is excluded; (2) `*.server.ts` naming convention — Start's import-protection fails the build if a client file imports a server-only module; (3) `ssr.external: ["better-sqlite3","@napi-rs/keyring","@junction/core"]` in `vite.config.ts` — defence-in-depth. Run the client-bundle leak grep after every web build: `grep -rl "better-sqlite3\|napi-rs/keyring\|CREATE TABLE\|drizzle" packages/web/dist/client` → must be empty.

- **jscpd excludes `data.server.ts` — intentional cross-app replication, not accidental duplication.** `packages/web/src/server/data.server.ts` intentionally replicates two patterns from CLI commands (`credentialStoreLabel` from `status.ts`, profile-source join loop from `profile.ts`). Cross-app imports (cli → web or web → cli) are banned; extracting to `core` would add display-policy logic to the data layer. Per the project's DRY policy, duplication between exactly 2 sibling apps is permitted until the rule of three triggers. The file is listed in `.jscpd.json`'s `ignore` to suppress the clone report. When this logic appears in a third app, extract it to core and remove the ignore entry. (inc 22)

- **`junction web` logged a TanStack `notFoundError` warning per unmatched request because no not-found component was configured.** Symptom: repeated `A notFoundError was encountered on the route with ID "__root__", but a notFoundComponent option was not configured…` on startup — one line per request that resolves to a not-found (browser `/favicon.ico` probe, mistyped path), so opening the dashboard with `--open` printed ~3. Not a crash; routes all serve 200. **Fix in place:** `defaultNotFoundComponent: NotFound` on `createRouter` (`router.tsx`) renders a real 404 (`components/not-found.tsx`) inside the root shell instead of TanStack's generic `<p>Not Found</p>`, AND an inline-SVG-data-URI `<link rel="icon">` in `__root.tsx`'s `head` stops the browser's `/favicon.ico` probe. The favicon **must** be inlined (or otherwise SSR-served) — `serve.mjs` only forwards to `app.fetch` and serves NO static files, so a `public/favicon.ico` would 404. Verify: warnings == 0 after hitting `/nope` + `/favicon.ico` against the rebuilt server. (inc 22)

- **The `*.test.ts` "built bin, child process" suites flake under parallel `pnpm verify` load (5s timeout).** Symptom: a spawned-binary test (e.g. `profile.test.ts > profile show --name …`) times out at ~5013ms in full `pnpm verify`, but passes clean when its file is run alone (`npx vitest run packages/cli/src/commands/profile.test.ts`). Cause: these tests spawn `node packages/cli/dist/index.js` and get CPU-starved past their 5s budget when all packages' suites run concurrently — not a logic failure. **Workaround:** re-run the single file to confirm green; do not "fix" by editing the test logic. If it becomes chronic, bump the per-test timeout for the child-process suites or mark them to run serially. (surfaced inc 22)

## CLI source / exec.ts (inc 21)

- **`spawnSandboxed` had no output byte cap — a flooding child OOMed junction.** A child that writes unbounded stdout/stderr to `stdoutChunks`/`stderrChunks` (`Buffer[]`) would grow the heap without limit; a path that runs `yes` or a flood script against a CLI source had no protection. **Fix (inc 21):** a shared `outBytes` counter tracks combined stdout+stderr across both stream handlers. When `outBytes > SPAWN_OUTPUT_BYTE_CAP` (1 MB), both streams set `outputCapped = true`, call `killGroup()`, and skip the chunk — exactly mirroring the timeout `killGroup()`. The result is `Ok(SandboxResult)` with `outputCapped: true` and truncated output (not an error; the child's partial output is returned). The cap is exported as `SPAWN_OUTPUT_BYTE_CAP` so callers can surface the limit in their error strings.

## Sandbox (inc 8)

- **Seatbelt deny-all-read SIGABRTs every binary (exit 134) — UNLESS you import Apple's base profile.** A naive "deny default + hand-rolled system read-allowlist" crashes the loader (dyld + the dyld shared cache need reads the enumeration always misses; the tell is exit **134** crash vs **1** clean denial). inc 8 worked around it with a broad `(allow file-read*)` + explicit denies (so reads were confined ONLY to ~/.junction). **RESOLVED in inc 21 — true read confinement IS achievable:** `(deny default)` + `(import "/System/Library/Sandbox/Profiles/bsd.sb")` gives the loader exactly the system reads it needs (Apple maintains that list across OS versions), and you then `(allow file-read* (subpath …))` ONLY for `readPaths` + cwd + the binary's own dir. Result: reads are deny-default; `~/.ssh`, `/etc/hosts`, `/tmp/other` are denied while readPaths work, no SIGABRT (verified on Darwin 27). The piece inc 8 lacked was the **bsd.sb import** — do not hand-roll the system allowlist. (Also: the binary dir must be in the allow set or a non-system/Homebrew binary won't load.)
- **bubblewrap leaked the credential into argv via `--setenv KEY VALUE`.** bwrap's `--setenv` puts the VALUE on bwrap's own command line → readable via `/proc/<pid>/cmdline` and `ps` for the child's lifetime — a real secret exposure on Linux (invisible in macOS QA, where Seatbelt passes env out-of-band). **Fix (inc 21):** drop `--clearenv` + the per-key `--setenv`; spawn bwrap with an explicit `env:` (Node's `env` REPLACES the environment — no `process.env` inheritance) so bwrap forwards exactly that scrubbed allowlist to the child. Credential values reach the child via the environment, never argv. Regression guard: `buildBwrapArgv` test asserts no secret value appears in the argv.
- **Seatbelt does NOT scrub environment.** A sandboxed child inherits the parent's env by default → `JUNCTION_MASTER_KEY` would leak straight through. **Fix:** the harness spawns children with an **explicit `env:` allowlist** built only from `policy.env`; never inherit `process.env`. (Verified: explicit env → child sees the secret empty.)
- **Deno permissions are a capability boundary, not a syscall jail.** `--allow-run`, `--allow-ffi`, `eval`, dynamic `import()`, and workers all escape or inherit privileges. **Fix:** always `--no-prompt --deny-run --deny-ffi`; for truly hostile code use the microVM tier, not Deno.
- **bubblewrap needs unprivileged user namespaces.** Some distros disable `CLONE_NEWUSER`. **Fix:** probe at runtime (`bwrap --ro-bind /usr /usr -- /bin/true`); if it fails, treat the platform as having no command sandbox → **refuse** (`Err`), never raw-exec.
- **The cardinal rule:** when no enforceable backend is available, **refuse** — never fall back to `child_process.spawn` unsandboxed. Silent unsandboxed fallback is the #1 way this layer becomes security theater.
- **SBPL/argv injection via unsanitized policy paths.** A granted path was interpolated raw into the Seatbelt profile (`(allow file-write* (subpath "${p}"))`) — a path containing `")) (allow …` injected *new* policy rules and escaped the workspace (proven: wrote outside, exit 0). The sandbox is *designed* to take semi-trusted input, so this is unsafe-by-construction. **Fix:** central `validatePolicy` rejects any path/cwd/allowNet entry containing SBPL/argv metachars (`"` `\` `(` `)` newline NUL `,`) → `policy-invalid`, before any profile/argv is built. Same metachar guard also blocks Deno's comma-list widening.
- **bwrap as a bare command name fails under a scrubbed env (Linux only).** `spawn("bwrap", …, {env:{}})` has no `PATH`, so Node can't resolve the bare binary → ENOENT → probe false → backend silently `"none"` → **bwrap tests skip**. Seatbelt avoided it by using the absolute `/usr/bin/sandbox-exec`. **Fix:** resolve bwrap to an absolute path via `which` once (like deno), use it for all spawns. Also: `resolveCapabilities` must actually *probe* the bwrap backend on Linux (it initially only probed seatbelt). **Lesson: a backend that only runs on a platform your dev box isn't must be exercised in CI** — local `verify` can't see it; the "tests silently skip" failure mode hides until a Linux runner proves they *ran* (assert the backend banner + non-skipped count).

## Credentials (inc 6, 13)

- **`removeCredential` swallows a `store.delete` failure after a successful DB delete (reverse-orphan).** If the DB row is deleted but the store delete errors, the secret becomes unreachable and stays in the store with no key to find it. The operation returns `Ok` (DB is the authority; a stranded store entry is preferable to surfacing a misleading error). **Symptom:** the DB credential is gone, the store entry is orphaned, no error or log. **Fix in place:** the best-effort `.orElse` swallow is intentional and tested (see `"reverse-orphan"` test in `repositories.test.ts`). **Forward action:** once pino lands, emit a `warn` log from `removeCredential` on store-delete failure so the orphan is observable in the audit log. (inc 13)

- **scrypt throws at `N=2^17` under the default 32 MiB `maxmem`** (`ERR_CRYPTO_INVALID_SCRYPT_PARAMS`). **Fix:** pass `maxmem: 256 * 1024 * 1024` explicitly. A test using weak params would miss this — test with the real params.
- **Corrupt-but-present `credentials.enc.json` must not be silently emptied.** Treating a JSON `SyntaxError` as "empty store" lets the next `set()` overwrite and destroy all ciphertext. **Fix:** only `ENOENT` → empty map; a present-but-unparseable file → `io-failed` (refuse to load/overwrite).
- **Temp files: create at `0600`, don't `chmod` after.** Writing the master key / salt tmp at the umask default then chmod-ing leaves a world-readable window on the actual key. **Fix:** `writeFile(tmp, data, { mode: 0o600 })`.

## MCP client — upstream connector (inc 11)

- **`env` on `StdioClientTransport` REPLACES the default environment.** A custom `env` object passed to `StdioClientTransport` is used verbatim — it does NOT inherit from `process.env`. If you omit `PATH` and `HOME`, the child process cannot resolve binaries or find its home directory and will fail with ENOENT or unexpected errors. **Fix:** always spread `getDefaultEnvironment()` (returns HOME, LOGNAME, PATH, SHELL, TERM, USER) first, then inject the token: `{ ...getDefaultEnvironment(), [tokenEnvVar]: secret }`. This also keeps the child env minimal — no `process.env` spill — which matches the sandbox env-scrub discipline (no `JUNCTION_MASTER_KEY` leaking to an upstream binary).
- **UpstreamError.cause leak-safety depends on `String(cause)` staying opaque to Error properties.** `formatUpstreamError` in `debug.ts` formats `cause` via `String(cause)` → `"Error: <message>"`, which does NOT include non-enumerable Error properties (e.g. axios's `.config.headers.Authorization`). This is the primary guarantee that a Bearer token in an axios-style error never surfaces on stdout/stderr. **Revisit if** the MCP SDK swaps its HTTP client away from `node-fetch`/`undici` to axios or a library that puts credentials in enumerable `.config.headers` on error objects — audit `formatUpstreamError` then to ensure `String(cause)` or a scrubbed serializer is used, never `JSON.stringify(cause)`. (Raised inc 11; `connect.test.ts` (g) is the regression guard.)

## MCP (inc 7)

- **A zero-tool `McpServer` doesn't advertise the `tools` capability** → a client's `tools/list` returns `-32601 Method not found`, not an empty list. **Fix:** use the low-level `Server` with `capabilities:{tools:{}}` + an explicit `ListToolsRequestSchema` handler returning the (profile-driven) tool list — which is also the right model for junction.
- **`stdout` on `junction mcp serve` is the MCP channel** — a single stray log line corrupts the JSON-RPC stream. **Fix:** all human/error output goes to **stderr**; never `console.log`/consola to stdout in the serve path.

## TUI (inc 9)

- **OpenTUI's native renderer is Bun-only under Node.** `@opentui/core` imports cleanly but calling `createCliRenderer()` or `createTestRenderer()` throws `"native FFI is not available for this runtime yet"` on Node 22/24 — the renderer uses `bun:ffi` internally, which is unavailable outside Bun. **Symptom:** import succeeds, first render call throws. **Fix:** use **Ink** for Node TUIs. Ink 7.x (7.1.0 here, `engines: node >=22`) is pure ESM, renders entirely in Node, and has `ink-testing-library` for headless snapshot tests. The OpenTUI revisit-when entry tracks the trigger. (inc 9)

## Boundary tooling (inc 1.5 / 3 / 7)

- **depcruise "green but blind":** resolving `@junction/*` through the `exports` map landed edges on the excluded `dist/`, so cross-package violations were silently invisible while the rules "passed". **Fix:** `tsconfig.depcruise.json` maps every specifier to `packages/*/src`, and the `cli` package (named `junction`, not `@junction/cli`) is mapped too so its boundary is governed.
- **Boundary rules must be structural, not enumerated.** Listing today's packages by name leaves a hole the moment a 6th package is added (it falls through the guard). **Fix:** "a lib = any non-app package may import only core + itself" — a new package is automatically governed. Re-verify a planted-import matrix on any rule change; the guard has needed guarding twice.

## Config / CLI (inc 2 / 3)

- **`proper-lockfile` with `realpath:true` ENOENTs on first write** (the target file doesn't exist yet). **Fix:** lock the **home directory**, not the not-yet-created file.
- **Temp filename via `Date.now()` collides under same-millisecond concurrency** → rename race → write-failed. **Fix:** `randomUUID()` for temp names.
- **`process.exit()` truncates a `--json` write on a pipe.** **Fix:** set `process.exitCode` and `return`; let the event loop flush stdout.
- **TS exhaustiveness `never`-guard via an `if`-chain (`const _x: never = e`) can break a clean `tsc -b` build** under some TS versions. **Fix:** use a `switch` with the `never` assignment in the `default` case.
- **⚠️ MYTH (do NOT "fix" this): "TS6 doesn't narrow switch-default to `never`, use `as never` / add a `default`".** This is FALSE and has caused the same regression three times (inc 20, 21). TS 6.0.3 DOES narrow an exhaustive discriminated-union switch's `default` to `never` — `const _: never = e` (no cast) and a defaultless exhaustive switch BOTH compile cleanly (verified on the pinned 6.0.3; `main` ships them and CI is green). The "phantom TS2322/TS2366" you see right after editing across packages is the **stale `.tsbuildinfo`** pattern (above), NOT a TS6 change. **Fix: run `pnpm build`** (forces a fresh cross-package recheck) — never add a non-exhaustive `default` and never weaken the guard to `const _: never = e as never` (the cast permanently defeats exhaustiveness, so a future-added union member silently falls through instead of failing the build). If you're tempted to add a `default` to `formatUpstreamError`/`formatConfigError`/`safeUpstreamMessage` etc., STOP — rebuild instead.

## OpenAPI provider (inc 15)

- **`@scalar/openapi-parser` API is mixed sync/async.** `validate()` is async (returns `Promise<ValidateResult>`); `dereference()` and `upgradeFromTwoToThree()` are synchronous. Getting this backwards causes either unresolved promises or calling `.then` on a non-promise. **Fix:** `await validate(doc)`; call `dereference(doc)` and `upgradeFromTwoToThree(doc)` without `await`.
- **apiKey-in-query: the request URL contains the secret.** The composed `URL.toString()` (with the key in its query string) must NEVER appear in a tool result, error message, log line, or any string derived from the response. **Fix:** `callOperation` returns only `"${status} ${statusText}\n${body}"` — no URL. The URL object is a local variable that leaves no trace in any output. Additionally, `fetch()`'s `TypeError` on a malformed URL embeds the full URL (with the key) in its `.message`; a pre-fetch `new URL(fullUrl)` check catches this and returns a sanitized `call-failed` cause, and the outer catch always uses `cause.constructor.name` instead of `cause.message`. (inc 15, hardened)
- **apiKey-in-query: the upstream API may REFLECT the secret in its response body** (e.g. an OAuth server echoing back query parameters). Junction returns upstream response bodies verbatim — this is inherent, the operator's choice. Defense: prefer `in: "header"` for API keys; if `in: "query"` is required, the operator must audit the upstream's response behaviour. Junction itself never inserts the key into the result, but cannot strip what the upstream inserts.
- **Slowloris / body-stall guard.** A malicious or slow upstream can send HTTP headers instantly then dribble the body byte-by-byte indefinitely. The pre-inc-15 code cleared the 30 s abort timer immediately after headers arrived, leaving the body-stream read without a deadline. **Fix (inc 15 hardened):** the timer is now kept armed through the full body read by moving `clearTimeout` into a `finally` block. `controller.abort()` fires at the timeout, `reader.read()` rejects with `AbortError`, and the existing catch maps it to `timed-out`.
- **Spec fetch had no byte cap.** `res.json()` on an unbounded spec endpoint would buffer the entire response into memory → OOM on multi-GB responses. **Fix (inc 15 hardened):** spec fetch is now a streaming read with a 10 MB cap (`SPEC_BYTE_CAP`). Exceeding it aborts the read and returns `spec-fetch-failed` with a "cap" cause.
- **OpenAPI 3.0 `nullable:true` is not valid JSON Schema.** JSON Schema doesn't have `nullable`; passing a 3.0 schema field with `nullable:true` unchanged to an agent will make validators reject the input. **Fix:** `normalizeSchema` in `tools.ts` recursively converts `{ type: T, nullable: true }` → `{ type: [T, "null"] }` (deleting `nullable`) before the schema is returned as `ProviderTool.inputSchema`.
- **`@scalar/openapi-parser`'s `dereference()` only resolves refs available in the in-memory document.** Remote `$ref` URLs (e.g. `"$ref": "https://…/types.json"`) are NOT fetched — they remain unresolved stubs. **Fix accepted:** we use `dereference` in "offline" mode intentionally (no remote fetching during tool calls). Pinned by regression test (`remote-$ref isolation` in `http-exec.test.ts`). If a spec uses external $refs, tools may have incomplete schemas. Document at `platform add` if verbose mode is added in inc 16.

## OpenAPI base-URL resolution (inc 18)

- **Relative `servers` URL resolution belongs at `platform add` time, not at serve/call time.** The spec's origin (its fetch URL) is only known during `platform add`. Once the spec is cached inline (`{from:"inline", document}`) the origin is lost, so the runtime `resolveBaseUrl` in `http.ts` can only resolve against `connection.baseUrl` or an *absolute* `servers[0].url`. **Fix:** `resolveSpecBaseUrl` runs at `platform add`, stores an absolute `baseUrl` in the descriptor. The runtime path is unchanged — it remains the fallback for platforms added before this increment (they keep working via the absolute-servers runtime path or an explicit `--base-url`).
- **vitest "green but blind" via missing package aliases (FIXED inc 18).** `vitest.config.ts` aliases `@junction/*` to source so `pnpm verify` tests src, not the built `dist/` (the packages' `exports` point at dist, which verify doesn't build). But the alias map originally listed only `@junction/core` + `@junction/mcp-server` — **`@junction/mcp-client` and `@junction/openapi-client` were missing**. So any cli test importing those by package name (`platform.ts`'s dynamic import, `providers.ts`, the OpenAPI debug-call test) silently ran against **stale/absent dist** — a real correctness blind spot (a src change with no rebuild → tests pass against old code), not mere friction. **Fix:** added both packages to the vitest `alias` map (→ `src/index.ts`). Proven by deleting both `dist/` dirs and running the cross-importing cli tests: still green = resolving to src. **Lesson:** every `@junction/*` package a test can import must be in the vitest alias map; when a new lib package is added, add its alias in the same change.
- **Server-variable templating (`{host}`, `{basePath}`, etc.) is not substituted.** OpenAPI 3.x allows templates like `https://{host}/v1` in `servers[0].url`. Junction doesn't substitute these — `resolveSpecBaseUrl` returns `base-url-has-variables`, requiring the operator to pass `--base-url`. If demand arises, a later increment can add a `--base-url-var` flag. (inc 18)

## OpenAPI selection + refresh (inc 19)

- **Selection must be persisted AND enforced at runtime — not just at add-time.** The runtime provider re-extracts tools from the full cached spec on every `listTools` call (`provider.ts`). A selection applied only at `platform add` is ignored at serve/debug time — the full over-cap set leaks. **Fix:** `select` is stored in the `OpenApiConnection` descriptor, and `createOpenApiProvider.listTools` passes `connection.select` to `extractTools`. Proven by the `provider-selection.test.ts` runtime-enforcement suite. If you ever see `too-many-tools` at serve time for a platform added with `--tag`, this is the invariant to check first.
- **Path-prefix matching needs a path-boundary check.** A naive `path.startsWith(prefix)` would match `/pets-shop` when the prefix is `/pet` (since `/pets-shop`.startsWith(`/pet`) is true). The correct boundary check: match only if the path equals the prefix exactly, or the next character after the prefix is `/`. Implemented in `operationMatchesSelection`. Any future path-filtering logic should replicate this check.
- **`platform refresh` must never clobber a working platform.** If the re-fetched spec would exceed the cap (after selection), refresh REFUSES and leaves both the DB descriptor and the cached spec file intact. A failing fetch also leaves everything unchanged. The no-clobber is enforced by returning before the `writeFile`/`upsert` calls — the test `"refreshed spec over cap → refuses, leaves DB descriptor and cache file unchanged"` is the regression guard.
- **`tsc -b` incremental cache can hide new function signatures.** When `extractTools` gained its `select?` third parameter, the first `tsc -b` run after the edit reported "Expected 1-2 arguments, but got 3" because the openapi-client's old `.tsbuildinfo` marked it as up-to-date and skipped recompilation. **Fix:** run `pnpm build` to force a tsdown rebuild of all packages, then `pnpm verify`. If typecheck fails with "too many arguments" (or, conversely, *phantom* exhaustiveness errors like "X is not assignable to never" / "function lacks ending return" on files you didn't touch) right after editing across packages, a stale `.tsbuildinfo` is the prime suspect.
- **Do NOT "fix" stale-buildinfo phantom errors with `tsc -b --force`.** `tsc -b --force` *emits* compiled JS into the same `packages/*/dist` that **tsdown** owns, clobbering the bundled runnable `cli/dist/index.js` with a tsc-compiled version that doesn't resolve the workspace deps the same way. Symptom after a `--force`: every "built bin, child process" test (credential/profile/platform `.test.ts` that spawn `node packages/cli/dist/index.js`) fails with "Command failed", even though unit tests (which run against `src` via the vitest alias) pass. **Fix:** clear the phantom errors by editing-then-rebuilding normally — `pnpm build` (tsdown) restores the proper dist; never `--force`. The child-process smoke tests always require a fresh `pnpm build` dist (they are the one suite that exercises the real bundled bin, not src).

## Optional credentials (inc 16)

- **Dropping a `NOT NULL` constraint on SQLite is a table REBUILD, not an additive migration.** Making `source_refs.credential_id` nullable can't be an `ALTER COLUMN` (SQLite has none) — drizzle-kit generates a `__new_source_refs` create + `INSERT…SELECT` + drop + rename (`0004_neat_spirit.sql`). **Always generate it via `pnpm drizzle-kit generate`, never hand-author** — a hand-authored migration misses `meta/NNNN_snapshot.json`, and the next generate then re-emits the whole table → duplicate-column failure (the snapshot-required gotcha, reaffirmed). The rebuild silently relies on the `INSERT…SELECT` column list exactly matching and on `source_refs` being a leaf table (nothing refs it) so the drop/rename orphans nothing.
- **A non-monotonic migration-journal `when` SILENTLY SKIPS later migrations on EXISTING DBs.** The drizzle better-sqlite3 migrator does NOT apply by `idx` order — it takes `lastApplied = SELECT … ORDER BY created_at DESC LIMIT 1` and applies only migrations whose `when` (folderMillis) is **>** that high-water. The hand-authored `0003_add_openapi_column` has an inflated `when` of `1782600000000` (a round, human-typed value ≈ today) that is LARGER than 0004/0005/0006. So a DB last migrated to 0003/0004 (created during inc 15–20) has its high-water stuck at `1782600000000`, and 0005 (`…507…`)/0006 (`…511…`) are silently skipped on upgrade → "no such column: graphql/cli" at runtime. (Correction: an earlier inc-16 note that this ordering was "harmless, applies by idx" was WRONG — it applies by the `created_at` high-water.) **Why it's currently low-impact:** fresh installs apply all migrations from scratch (no high-water to trip), and every *future* migration gets a real `drizzle-kit generate` timestamp > today's, so it exceeds the stale value and applies — only DBs born in the inc 15–20 window are stuck, i.e. dev/test homes (no distributed users yet). **Recreate** a stuck dev DB (`rm -rf <home>`). The journal-normalization + a regression test are deferred to the distribution increment (see `revisit-when.md`). **Rule reinforced:** never hand-author a migration (timestamp included) — always `drizzle-kit generate`, which keeps `when` monotonic.
- **A migration test that seeds *after* the migrations run proves nothing about data preservation.** The `repositories` suite migrates straight to HEAD (`getDatabase`), so seeding there only exercises the post-rebuild schema — it cannot prove the rebuild *preserved* pre-existing rows ("green but blind"). **Fix in place:** a separate `migration 0004 — cross-version data preservation` block applies `0000→0003` on a raw `better-sqlite3` DB, seeds a credentialed source the old way, *then* applies `0004` and asserts the row + RESTRICT FK + unique index survived and that NULL now inserts. The cross-version staged shape is the only honest preservation proof. (Note: `readFileSync` is boundary-guard-blocked even in core tests — use async `readFile`.)
- **`platform add --kind mcp --transport http` always materializes an `auth` descriptor**, even when none was requested. Consequence for inc 16: a *public* http-MCP source (no credential) will **always** trip the new "platform declares auth but no credential attached" warning on `mcp serve`, because `connection.auth` is never undefined for http. The warning is stderr-only and non-blocking (an http-MCP platform genuinely *can* take a bearer token, so the warning isn't wrong), but it's unavoidably noisy for this class. Root cause is the platform-add default, not the warning. **Forward action:** when `platform add` gets auth-shape polish (inc 17+), stop defaulting `auth` for http-MCP unless a scheme is requested, which also makes this warning precise.

## Testing

- **"Built bin, child-process" tests are flaky at the 5s default timeout under load.** Several suites (`cli/src/commands/{profile,credential,platform,debug}.test.ts`) spawn `node packages/cli/dist/index.js` and do real work. Run standalone they pass in well under 5s; but when the whole suite runs concurrently (e.g. the lefthook **pre-push** `verify` on a loaded machine), individual child-process tests exceed 5s and time out → `git push` fails on a *flaky* gate, not a real regression (passes on standalone re-run). **Fix:** `testTimeout`/`hookTimeout` = 20s in `vitest.config.ts` (headroom without masking truly-hung tests). Tell: the failures are `Test timed out in 5000ms` on child-process suites, and they pass when re-run alone.

## GraphQL provider (inc 20)

- **`drizzle-kit generate` is not available at the monorepo root** — `drizzle-kit` is only installed in `packages/core/node_modules`. Running `pnpm drizzle-kit generate` from root fails with "not found". **Fix:** write a temp config pointing at `packages/core/src/db/schema.ts`, then run `tsx node_modules/drizzle-kit/bin.cjs generate --config /path/to/temp.config.ts` from inside `packages/core/` using its local node_modules. Always verify the generated snapshot lands in `packages/core/src/db/migrations/meta/`.
- **GraphQL introspection returns raw `__schema` JSON — `buildClientSchema` requires it wrapped as `{ data: { __schema } }`.** Passing the unwrapped object throws `"Cannot read properties of undefined"`. **Fix:** the introspection query body is `{ data: { __schema } }` (standard GraphQL response envelope); always pass `result.data` to `buildClientSchema`, not the top-level object.
- **`parse("")` (empty string) throws a syntax error, not returning an empty document.** An empty `query` string sent to `graphql_query` or `graphql_mutation` should return `invalid-args`, which it does — but the code path goes through the `try/catch` around `parse`, not through the "no operation definitions" branch. Tests must cover empty input explicitly.
- **TypeScript 6.x array-index narrowing: `defs[0]` is typed as `T | undefined` even after a `defs.length === 1` guard.** The narrowing that "defs has exactly one element therefore defs[0] is defined" is not emitted by TS6 strict mode. **Fix:** cast `defs[0] as OperationDefinitionNode` at the guarded return site. The guard above proves safety; the cast is load-bearing for compilation.

- **SSR request data in TanStack Start: use `getRequest()`/`getCookie()` from `@tanstack/react-start/server`, never a fabricated global.** (raised inc 23 Phase E) The sidebar SSR cookie read invented `globalThis.__tss_request_headers` (does not exist; an `as any` cast hid it from tsc) → the server always rendered the default, defeating the no-flash mechanism. The real APIs are `getRequest()` (returns the `Request`; already used by `data.functions.ts` host guard) and the `getCookie(name)`/`getRequestHeader(name)` helpers from the same module. **Also:** an SSR "no-flash" pre-hydration attribute is only real if a CSS selector actually consumes it — `data-sidebar` was set on `<body>` but no `[data-sidebar=...]` rule existed, so it did nothing. And drive layout from ONE source of truth: a single `[data-sidebar]` attribute → a `--sidebar-current` CSS var that BOTH the sidebar width and the content margin read, so they can't desync (the original had React state for width + a static prop for margin → ~192px dead gap on collapse). Symptom that hides it: build + unit tests + a casual screenshot all pass ("green but blind") because tests seed `document.cookie` client-side and never exercise the SSR/offset paths.

- **`pnpm verify` historically did NOT run `vite build` — fixed inc 23 dev-hardening; `verify` now runs `verify:web` (build + leak-check + smoke).** (raised inc 23 Phase E, resolved inc 23 dev-hardening) Originally `verify` = docs:check + tsc + Biome + tests + web typecheck — NO web build, so a TanStack-Start import-protection failure (e.g. importing `@tanstack/react-start/server` into a client-graph route file like `__root.tsx`) was invisible locally (`verify` green while `vite build` failed). Now `verify` chains **`verify:web`** (builds core+web, `web:leakcheck`, `web:smoke` against the running server), so the build break + dead-SSR class fails the local gate. Still good to remember: tests seed `document.cookie` client-side and never exercise the real SSR path — the smoke test does. Sanctioned way to use request data (cookies/headers) from a route: a `createServerFn` (server import stays in the server graph) called from `beforeLoad`, NOT a direct `getRequest()` in the route module. See `data.functions.ts:getSidebarState`.

- **tsdown config-loader needs `unrun` on Node < 22.6 — `pnpm build` was silently Node-20-broken; FIXED by adding `unrun`.** (raised + resolved inc 23 dev-hardening) tsdown's `auto` config-loader loads each package's `tsdown.config.ts`; on Node ≥22.6 it uses native TS, but on **Node 20** it requires the **`unrun`** loader — which wasn't installed, so every tsdown build (`core`, `cli`, `openapi-client`, `graphql-client`, mcp/*) failed on Node 20 with "Failed to import module 'unrun'". Latent the whole time (floor is Node 20, but `pnpm build` had never run on Node 20 in CI until `verify:web` did). **Fix: added `unrun` as a root devDependency** so the auto-loader finds it on Node 20; floor stays 20. (Rejected: `--config-loader tsx` corrupts the chained `node` migrations-copy step via tsx's namespace hook → `node:fs?tsx-namespace` ENOENT; `--config-loader native` needs Node ≥22.6.) `verify:web` is therefore back in the matrix `verify` (build break caught locally on Node 20 + 22). Re-confirm the loader behaviour if tsdown is upgraded.
